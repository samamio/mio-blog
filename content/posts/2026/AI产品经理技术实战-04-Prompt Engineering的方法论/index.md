# 第4章 让AI"说人话"——Prompt Engineering的方法论


---

## 一、背景：为什么Prompt是一门"工程"？

### 1.1 一个常见的误解

很多产品经理第一次接触大模型时，会这样写Prompt：

> "总结一下这篇文章。"

然后惊讶地发现，模型给出的摘要要么太长，要么太短，要么漏掉了关键信息。

他们的第一反应是："这个模型好笨。"

但真相是：**不是模型笨，是Prompt写得不好。**

同样的请求，换一个写法：

> "你是一个技术文档编辑。请阅读下面的文章，提取3个核心观点，每个观点不超过50字。输出格式为编号列表。"

结果会完全不同。

### 1.2 Prompt Engineering为什么需要"工程"思维？

因为Prompt不是"一句话"，而是一个**系统设计**。

一个高质量的Prompt包含多个组成部分：

```
1. 角色设定：告诉模型"你是谁"
2. 任务描述：告诉模型"你要做什么"
3. 约束条件：告诉模型"什么不能做"
4. 输出格式：告诉模型"结果长什么样"
5. 示例（可选）：给模型看几个"好的答案"
```

这跟工程设计一个系统是一样的思路：**明确输入、明确处理规则、明确输出**。

### 1.3 Prompt Engineering在AI产品中的地位

如果把AI产品比作一家餐厅：

- **模型**是厨师——他有能力做各种菜
- **Prompt**是菜谱——告诉厨师具体怎么做
- **RAG**是食材仓库——给厨师提供原材料
- **Function Calling**是厨房设备——让厨师能实际操作

再厉害的厨师，如果没有好的菜谱，也做不出好菜。再好的食材，如果没有好的菜谱，也做不出好菜。

**Prompt Engineering是连接模型能力和产品体验的桥梁。**

## 二、核心原理：Prompt的设计方法

### 2.1 Prompt的本质：引导而非命令

首先要纠正一个观念：**Prompt不是命令，而是引导。**

你给模型写Prompt，不是在"告诉它该做什么"，而是在"帮它理解你想要什么"。

类比：

- 命令式思维："给我总结这篇文章"——像是给下属下命令
- 引导式思维："请阅读下面的文章，提取三个最重要的观点，每个观点用一句话概括"——像是给同事做 briefing

引导式思维之所以更有效，是因为它给了模型更多的**上下文**和**约束**。

### 2.2 四段式Prompt框架

这是我推荐的Prompt设计框架，包含四个部分：

#### 第一部分：角色设定（Role）

> "你是一个资深客服专家，拥有10年的客户工单处理经验。"

**为什么重要：** 角色设定会影响模型的"语气"和"专业度"。让模型扮演一个专家，它会输出更专业的内容。

**常见角色设定：**
- 你是一个资深的技术文档编辑
- 你是一个专业的客服代表
- 你是一个经验丰富的产品经理
- 你是一个严谨的数据分析师

#### 第二部分：任务描述（Task）

> "请阅读下面的用户工单，判断它属于以下哪个类别：技术故障、功能建议、账号问题、其他。"

**为什么重要：** 任务描述是Prompt的核心。它告诉模型具体要做什么。

**好的任务描述应该：**
- 动词开头（请分析、请总结、请判断）
- 目标清晰（判断类别、提取关键信息）
- 范围明确（只分析正文，不包括标题）

#### 第三部分：约束条件（Constraints）

> "注意：如果工单内容不足5个字，请标记为'信息不足'。如果无法确定类别，请选择'其他'并说明原因。"

**为什么重要：** 约束条件防止模型"过度发挥"。没有约束的模型可能会输出你不需要的内容。

**常见约束类型：**
- 长度约束："不超过200字"
- 格式约束："用JSON格式输出"
- 行为约束："不要编造不存在的信息"
- 边界约束："只回答与问题相关的内容"

#### 第四部分：输出格式（Output Format）

> "输出格式如下：\n类别：[分类结果]\n置信度：[高/中/低]\n理由：[一句话说明]"

**为什么重要：** 统一的输出格式方便后续程序处理，也方便用户阅读。

**常见输出格式：**
- 编号列表
- JSON格式
- 表格
- 固定模板

### 2.3 完整的Prompt示例

让我们把四段式框架组合起来，看一个完整的Prompt：

```
【角色设定】
你是一个资深客服专家，拥有10年的客户工单处理经验。

【任务描述】
请阅读下面的用户工单，判断它属于以下哪个类别：
1. 技术故障 - 产品出现错误、崩溃、无法使用等问题
2. 功能建议 - 用户提出新功能或改进建议
3. 账号问题 - 登录、注册、密码、权限等问题
4. 其他 - 不属于以上三类的问题

【约束条件】
- 如果工单内容不足5个字，标记为"信息不足"
- 如果无法确定类别，选择"其他"并在理由中说明
- 不要添加类别列表之外的分类

【输出格式】
类别：[分类结果]
置信度：[高/中/低]
理由：[一句话说明]
```

### 2.4 三种推理触发器

#### Zero-shot（零样本）

**定义：** 不给任何示例，直接让模型完成任务。

```
请将以下评论分类为正面或负面：
"这个产品太难用了"
```

**适用场景：**
- 任务简单，模型应该能直接理解
- 没有足够的示例数据
- 快速验证想法

**优点：** 简单、快速
**缺点：** 效果不稳定，尤其是对复杂任务

#### Few-shot（少样本）

**定义：** 给模型看几个"好的答案"作为示例，让它模仿。

```
请将以下评论分类为正面或负面。

示例1：
评论："这个产品太好用了！"
分类：正面

示例2：
评论："客服态度很差。"
分类：负面

示例3：
评论："功能还行，但价格有点贵。"
分类：正面

待分类：
评论："这个产品太难用了"
分类：
```

**适用场景：**
- 任务有一定复杂度
- 你有标注好的示例数据
- 你需要模型模仿特定的输出风格

**优点：** 效果稳定，风格可控
**缺点：** 需要示例数据，Token消耗较多

#### Chain-of-Thought（思维链）

**定义：** 让模型在给出答案之前，先"想一步"——展示推理过程。

```
请判断以下工单的紧急程度，并说明你的推理过程。

工单："我的账号被锁定了，今天有一个重要的客户会议要用这个账号。"

思考过程：
1. 用户说账号被锁定 → 这是账号问题
2. 用户提到"重要的客户会议" → 有时间压力
3. 账号锁定 + 时间压力 → 紧急度高

紧急程度：高
```

**适用场景：**
- 任务需要推理（分类、判断、分析）
- 你需要模型给出有理有据的答案
- 任务有歧义，需要模型展示思考过程

**优点：** 答案更准确，可追溯推理过程
**缺点：** 输出更长，Token消耗更多

### 2.5 系统提示词 vs 用户提示词

在大模型API中，Prompt分为两类：

**系统提示词（System Prompt）**

- 由开发者设定，用户看不到
- 定义模型的整体行为和角色
- 一次设定，多次复用

```
系统提示词：你是一个专业的客服助手。你的任务是帮助用户解决问题。
如果问题超出你的能力范围，请礼貌地告知用户并转接人工客服。
```

**用户提示词（User Prompt）**

- 由最终用户输入
- 定义具体的任务和内容
- 每次请求都不同

```
用户提示词：我的账号登录不了了，提示密码错误，但我确定密码是对的。
```

**为什么区分这两者很重要？**

因为系统提示词定义了"游戏规则"，用户提示词是"在游戏内的具体操作"。一个好的系统提示词能让模型在所有用户交互中都保持一致的行为。

**产品经理的应用：**

- 系统提示词是你的"产品灵魂"——它定义了AI助手的性格和行为边界
- 用户提示词是你的"输入口"——它决定了用户怎么跟AI交互
- 系统提示词的权重通常高于用户提示词——即使用户说"忽略之前的指示"，系统提示词仍然有效

## 三、企业应用：Prompt Engineering的最佳实践

### 3.1 写好Prompt的10条原则

**原则一：具体胜过模糊**

- ❌ "总结一下"
- ✅ "提取3个核心观点，每个观点不超过50字"

**原则二：给角色不给头衔**

- ❌ "你是一个专家"
- ✅ "你是一个有10年经验的客服专家"

**原则三：正向描述胜过负向描述**

- ❌ "不要写太长的回答"
- ✅ "回答控制在200字以内"

**原则四：给示例胜过不给示例**

- ❌ "像这样分类"
- ✅ 给出3-5个分类示例

**原则五：分步骤胜过一步到位**

- ❌ "分析这段文本并给出建议"
- ✅ "第一步：提取文本中的关键信息。第二步：根据关键信息给出建议。"

**原则六：定义清楚输出格式**

- ❌ "给我一个结果"
- ✅ "输出格式：JSON，包含field_a和field_b两个字段"

**原则七：设定边界条件**

- ❌ "处理所有情况"
- ✅ "如果输入为空，返回'请提供文本'"

**原则八：用分隔符区分不同部分**

- ❌ 把所有内容混在一起
- ✅ 用```、---、###等分隔符区分不同部分

**原则九：控制Token消耗**

- ❌ 写一个5000字的Prompt
- ✅ 精简到必要部分，控制在2000字以内

**原则十：持续迭代**

- ❌ 写一遍就完了
- ✅ 每次效果不好时，分析原因，调整Prompt

### 3.2 Prompt版本管理

Prompt不是"写一次就用 forever"的。它需要像代码一样被管理。

**为什么需要版本管理？**

1. **效果会波动**：同一个Prompt，不同时期效果可能不同（因为底层模型可能更新）
2. **需求会变化**：产品需求变了，Prompt也要跟着变
3. **A/B测试**：你需要比较不同Prompt的效果
4. **回滚**：如果新版本Prompt效果变差了，你需要能回退到旧版本

**Prompt管理的最佳实践：**

```
prompt_v1.json  ← 2024-01-01 初始版本
prompt_v2.json  ← 2024-01-15 增加了few-shot示例
prompt_v3.json  ← 2024-02-01 调整了输出格式
prompt_v4.json  ← 2024-02-20 修复了幻觉问题
```

每个版本记录：

- 版本号
- 修改日期
- 修改原因
- 效果对比数据

**产品经理的应用：**

在你的PRD中，应该包含Prompt的版本信息和变更记录。不要只写"用大模型处理"就完了。

### 3.3 Prompt调试方法

当Prompt效果不好时，怎么调试？

**第一步：隔离变量**

每次只改一个地方。不要同时改角色、任务、约束三个部分——你不知道是哪个改动导致了效果变化。

**第二步：构造测试集**

准备10-20个典型的输入样例，覆盖各种常见场景和边界情况。用同一套测试集反复测试不同版本的Prompt。

**第三步：分析Bad Case**

对测试集中模型回答错误的案例，分析原因：

- 是角色设定不够清晰？
- 是任务描述有歧义？
- 是约束条件不充分？
- 是输出格式不合适？

**第四步：迭代优化**

根据分析结果调整Prompt，然后用测试集重新验证。

**第五步：线上监控**

上线后持续监控实际效果，收集新的Bad Case，回到第一步。

## 四、产品经理需要掌握什么？

### 4.1 Prompt工程师是什么角色？

在一些大公司，"Prompt工程师"是一个正式的岗位。他们的日常工作是：

1. 设计Prompt：根据产品需求，编写和优化Prompt
2. A/B测试：比较不同Prompt的效果
3. 维护Prompt库：管理不同场景的Prompt版本
4. 效果监控：跟踪Prompt的线上效果

但**作为产品经理，你不需要成为专业的Prompt工程师**。你需要的是：

1. 能写出合格的Prompt（四段式框架）
2. 能跟Prompt工程师有效沟通（知道要什么、为什么效果不好）
3. 能评估Prompt的效果（知道什么算好、什么算不好）

### 4.2 Prompt能力矩阵

| 级别 | 能力 | 典型场景 |
|------|------|---------|
| 入门 | 能写简单的指令型Prompt | "总结一下这篇文章" |
| 初级 | 能用四段式框架写结构化Prompt | 角色+任务+约束+输出格式 |
| 中级 | 能用Few-shot和CoT提升效果 | 给示例、让模型展示推理过程 |
| 高级 | 能系统化地管理和迭代Prompt | 版本管理、A/B测试、效果监控 |
| 专家 | 能设计Prompt框架和模板库 | 跨场景复用的Prompt组件 |

产品经理应该达到**中级水平**，能够独立完成大部分Prompt设计工作。

### 4.3 跟Prompt工程师沟通的常用话术

**当你需要Prompt工程师帮你优化时：**

> "这个Prompt在XX场景下效果不好，模型总是输出YY。我准备了10个测试用例，其中3个失败了。你能帮我看看是Prompt的问题还是模型的问题吗？"

**当你需要评估Prompt效果时：**

> "这个Prompt的准确率是多少？我用测试集测了20个case，准确率75%。你觉得要达到什么标准才能上线？"

**当你需要迭代Prompt时：**

> "上次上线的Prompt在用户反馈中发现一个问题：模型偶尔会编造不存在的知识库内容。我们能不能在Prompt里加一条约束，明确要求模型'只基于提供的知识库内容回答，不编造信息'？"

## 五、研发如何实现：Prompt的工程化

### 5.1 Prompt在系统中的位置

```

│                                              │
│  用户输入："我的订单到哪了？"                 │
└──────────────────────┬──────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────┐
│              Prompt组装器                    │
│                                              │
│  系统提示词（固定）                           │
│  +                                           │
│  用户提示词（动态）                           │
│  +                                           │
│  上下文/知识库（动态）                        │
│  =                                           │
│  最终发送给模型的Prompt                      │
└──────────────────────┬──────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────┐
│                 大模型API                    │
│                                              │
│  输入：组装好的Prompt                        │
│  输出：模型的回答                             │
└──────────────────────┬──────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────┐
│              Post-processing                 │
│                                              │
│  解析模型输出（JSON/文本）                    │
│  格式化结果                                  │
│  返回给用户                                  │
└─────────────────────────────────────────────┘
```

### 5.2 Prompt模板引擎

在实际系统中，Prompt不是硬编码的，而是用模板引擎动态组装的。

**模板示例（Python伪代码）：**

```python
system_prompt = """你是一个专业的客服助手。
你的任务是回答用户关于产品使用的问题。
如果问题超出你的能力范围，请转接人工客服。"""

user_prompt_template = """请根据以下知识库内容回答用户问题：

【知识库】
{knowledge_base}

【用户问题】
{user_question}

请只基于知识库内容回答，不要编造信息。"""

def assemble_prompt(user_question, knowledge_docs):
    knowledge_base = "\n".join(knowledge_docs)
    user_prompt = user_prompt_template.format(
        knowledge_base=knowledge_base,
        user_question=user_question
    )
    return system_prompt, user_prompt
```

**产品经理需要了解这个吗？**

不需要写代码，但需要理解：

1. Prompt是动态组装的——系统提示词固定，用户提示词根据输入变化
2. 知识库内容是动态插入的——这就是RAG的作用
3. 模板中有占位符（`{knowledge_base}`、`{user_question}`）——这些由系统填充

### 5.3 Prompt的存储和管理

在生产系统中，Prompt存储在数据库中，而不是代码里。这样做的好处是：

- **可以热更新**：改Prompt不需要重新发版
- **可以A/B测试**：同时运行多个版本的Prompt
- **可以灰度发布**：先对10%的用户用新Prompt

**数据库表设计示例：**

```
prompt_templates表：
+----+----------+-------------+------+------------------+
| id | name     | version     | type | content          |
+----+----------+-------------+------+------------------+
| 1  | 工单分类  | v3.2        | system | 你是一个客服专... |
| 2  | 工单分类  | v3.2        | user   | 请阅读以下工单... |
| 3  | 智能回复  | v1.5        | system | 你是一个回复助手... |
+----+----------+-------------+------+------------------+
```

## 七、真实案例：客户工单系统的Prompt迭代

### 7.1 场景一：工单自动分类

**需求：** 将工单自动分类为四类（技术故障/功能建议/账号问题/其他）

**第一版Prompt：**

```
请分类以下工单：
{工单内容}

分类选项：技术故障、功能建议、账号问题、其他
```

效果：准确率62%。问题：模型经常混淆"功能建议"和"技术故障"。

**第二版Prompt（增加角色和定义）：**

```
你是一个资深客服专家。请阅读以下工单，将其分类为四类之一：

- 技术故障：产品出现错误、崩溃、无法使用等问题
- 功能建议：用户提出新功能或改进建议
- 账号问题：登录、注册、密码、权限等问题
- 其他：不属于以上三类的问题

工单内容：
{工单内容}

请只输出分类结果，不要输出其他内容。
```

效果：准确率75%。改善了，但还不够。

**第三版Prompt（增加Few-shot示例和JSON输出）：**

```
你是一个资深客服专家，负责分类用户工单。

分类标准：
- 技术故障：产品出现错误、崩溃、无法使用等问题
- 功能建议：用户提出新功能或改进建议
- 账号问题：登录、注册、密码、权限等问题
- 其他：不属于以上三类的问题

示例：
工单："登录后页面一直在加载" → 技术故障
工单："希望能支持导出Excel" → 功能建议
工单："忘记密码了，怎么重置" → 账号问题
工单："你们的产品真好" → 其他

请分类以下工单：
{工单内容}

输出格式（JSON）：
{"category": "分类结果", "confidence": "高/中/低"}
```

效果：准确率88%。达标！

### 7.2 场景二：智能回复建议

**需求：** 根据工单内容和知识库，生成回复建议

**第一版Prompt：**

```
请回答以下用户问题：
{工单内容}
```

效果：很差。模型经常编造答案。

**第二版Prompt（加入RAG）：**

```
你是一个客服专家。请根据以下知识库内容回答用户问题。

【知识库】
{检索到的相关知识文档}

【用户问题】
{工单内容}

请只基于知识库内容回答，不要编造信息。
```

效果：好多了，但回复太生硬，像机器人在念文档。

**第三版Prompt（加入语气指导和格式要求）：**

```
你是一个专业的客服专家，语气友好、专业、简洁。

请根据以下知识库内容，为用户的工单生成回复建议。

【知识库】
{检索到的相关知识文档}

【用户工单】
{工单内容}

要求：
1. 先表达对用户问题的理解
2. 基于知识库给出解决方案
3. 如果知识库中没有相关信息，告知用户将转接人工
4. 回复控制在150字以内
5. 语气友好，不要使用过于技术化的术语
```

效果：回复质量显著提升，用户满意度从68%提升到82%。

### 7.3 Prompt迭化的关键经验

1. **示例的力量巨大**：从62%到75%，只加了定义。从75%到88%，加了Few-shot示例。
2. **约束条件不可少**：没有约束的Prompt，模型会"自由发挥"，结果不可控。
3. **语气和格式同样重要**：不只是"答对"，还要"答得好"——语气友好、格式统一。
4. **RAG是Prompt的"外挂大脑"**：Prompt告诉模型"怎么做"，RAG告诉模型"知道什么"。两者配合才能达到好效果。

## 八、常见误区

### 误区一："Prompt越長越好"

**真相**：Prompt不是越长越好。过长的Prompt会：
- 消耗更多Token（更贵）
- 增加模型的理解负担（可能忽略关键指令）
- 降低响应速度（更慢）

**正确做法**：精简到必要部分。能用100字说清楚的，不要用500字。

### 误区二："Prompt写好了就一劳永逸"

**真相**：Prompt需要持续维护。模型会更新、需求会变、用户行为会演化。一个三个月前的Prompt，现在可能已经过时了。

**正确做法**：建立Prompt的定期审查机制，至少每季度检查一次。

### 误区三："Prompt可以解决所有效果问题"

**真相**：Prompt只能优化"模型怎么用"，不能改变"模型能做什么"。如果模型本身就不擅长某个任务（比如精确计算），再好的Prompt也没用。

**正确做法**：先确认模型能力边界，再设计Prompt。不要在模型不擅长的任务上浪费精力。

### 误区四："所有场景都用同一个Prompt"

**真相**：不同场景需要不同的Prompt。客服场景和写作场景的Prompt差异很大。用一个通用Prompt处理所有场景，效果通常不好。

**正确做法**：为每个场景设计专门的Prompt模板，建立Prompt库。

### 误区五："Prompt就是用户说的话"

**真相**：用户说的话只是Prompt的一部分。完整的Prompt还包括系统提示词、上下文信息、知识库内容等。

**正确做法**：理解Prompt的完整结构，不要只关注用户输入部分。

### 误区六："Prompt效果不好就是模型不行"

**真相**：Prompt效果差，70%的情况是Prompt本身的问题，不是模型的问题。

**正确做法**：先优化Prompt，再考虑换模型。

### 误区七："Prompt是工程师的事，跟PM无关"

**真相**：Prompt是产品体验的核心组成部分。同一个模型，不同的Prompt，用户体验可能差十倍。产品经理需要对最终的用户体验负责，而Prompt是影响用户体验的关键因素。

**正确做法**：积极参与Prompt的设计和优化，不要完全丢给工程师。

## 九、面试题

### 初级

**Q1：什么是Prompt？为什么Prompt对AI产品的效果这么重要？**

**参考答案：**

Prompt是用户或系统发送给大模型的输入文本，用于告诉模型要做什么。

Prompt之所以重要，是因为：

1. 大模型本身是"通用"的——它什么都会一点，但什么都不精。Prompt的作用是把它"引导"到正确的方向。
2. 同样的模型，不同的Prompt，效果可能差好几倍。
3. Prompt定义了模型的"角色"、"任务"、"约束"和"输出格式"——这四个要素直接决定了输出质量。

类比：模型是厨师，Prompt是菜谱。再厉害的厨师，如果没有好的菜谱，也做不出好菜。

---

**Q2：请写一个Prompt，让模型把一段文字翻译成英文。**

**参考答案：**

一个合格的Prompt：

```
你是一个专业的翻译专家。请将以下中文文字翻译成英文，要求：
1. 保持原文的意思不变
2. 使用自然的英文表达
3. 不要添加原文中没有的内容

中文：{待翻译的文字}

英文翻译：
```

关键点：
- 角色设定（翻译专家）
- 任务描述（翻译成英文）
- 约束条件（保持原意、自然表达、不添加内容）
- 输出格式（直接给出翻译结果）

---

### 中级

**Q3：你的AI客服产品的回复质量不稳定，有时很好有时很差。你怀疑是Prompt的问题，你会怎么分析和解决？**

**参考答案：**

我会按以下步骤排查：

**第一步：收集数据**
- 导出最近一周的对话记录
- 标记出"质量好"和"质量差"的案例各20条

**第二步：分析差异**
- 质量好的案例：Prompt中哪些元素起了作用？（角色设定清晰？示例充分？）
- 质量差的案例：缺少了什么？（约束不够？没有示例？上下文信息不足？）

**第三步：定位Prompt问题**
- 检查系统提示词：是否定义了清晰的角色和行为边界？
- 检查用户提示词：是否包含了足够的上下文信息？
- 检查知识库检索：RAG检索到的文档是否与用户问题相关？

**第四步：设计A/B测试**
- 版本A：当前Prompt
- 版本B：优化后的Prompt（增加Few-shot示例、加强约束条件）
- 各跑1000条对话，比较效果

**第五步：持续监控**
- 上线新版本后，持续跟踪效果指标
- 收集新的Bad Case，继续迭代

---

**Q4：Zero-shot、Few-shot、Chain-of-Thought三种方法有什么区别？什么时候用哪种？**

**参考答案：**

**Zero-shot（零样本）**：不给示例，直接让模型完成任务。
- 适用：任务简单、模型应该能直接理解的场景
- 优点：简单快速、Token消耗少
- 缺点：效果不稳定

**Few-shot（少样本）**：给模型看几个示例，让它模仿。
- 适用：需要模型模仿特定风格或格式的场景
- 优点：效果稳定、风格可控
- 缺点：需要示例数据、Token消耗多

**Chain-of-Thought（思维链）**：让模型先展示推理过程，再给出答案。
- 适用：需要推理判断的复杂任务
- 优点：答案更准确、可追溯
- 缺点：输出很长、Token消耗最多

**我的选择原则**：
- 简单任务 → Zero-shot
- 需要格式/风格一致 → Few-shot
- 需要推理/判断 → Chain-of-Thought
- 不确定时 → 先用Zero-shot快速验证，效果不好再加Few-shot或CoT

### 高级

**Q5：你设计了一个Prompt，在测试集上准确率90%，但上线后只有70%。可能的原因是什么？怎么解决？**

**参考答案：**

可能的原因：

**1. 测试集不具代表性**
- 测试集可能偏向简单案例，线上有各种边界情况
- 解决：扩大测试集，覆盖更多真实场景

**2. 线上输入质量不如测试数据**
- 测试数据是干净的，线上用户输入可能包含乱码、表情符号、不完整句子
- 解决：在前端增加输入清洗和校验

**3. 模型版本更新了**
- 底层模型可能自动升级，导致Prompt效果变化
- 解决：每次模型升级后重新验证Prompt

**4. Prompt的上下文依赖了测试环境的数据**
- 测试时的知识库数据和线上的可能不同
- 解决：确保测试环境和线上环境的数据一致

**5. Prompt被部分截断**
- 线上输入的Token数可能超过Prompt的长度限制
- 解决：增加长度检测，超长输入时截断或分段处理

---

**Q6：如何系统地评估一个Prompt的质量？**

**参考答案：**

我会从四个维度评估：

**1. 准确性**
- 用标准化测试集（至少50条用例）评估
- 计算准确率、召回率、F1分数
- 按不同场景分别统计（不要只看总体）

**2. 稳定性**
- 同一输入跑多次，观察输出是否一致
- 计算输出的方差（对于生成任务）
- Temperature设0时应该完全一致

**3. 鲁棒性**
- 用边界输入测试（空输入、超长输入、特殊字符、多语言混合）
- 观察Prompt在这些情况下是否会"崩溃"
- 应该有适当的错误处理和降级策略

**4. 成本效率**
- 测量每个请求的Token消耗
- 计算单次调用的成本
- 在保证效果的前提下，尽量精简Prompt

**评估流程**：
1. 建立测试集（覆盖正常场景和边界场景）
2. 运行所有测试用例，记录结果
3. 分析Bad Case，定位Prompt问题
4. 调整Prompt，重新运行测试
5. 重复直到达到效果目标
6. 上线后持续监控，定期回归测试

## 十、本章总结

本章我们系统学习了Prompt Engineering的方法论：

1. **Prompt的本质**：不是命令，而是引导。好的Prompt给模型足够的上下文和约束。
2. **四段式框架**：角色设定 + 任务描述 + 约束条件 + 输出格式。这是Prompt设计的"最小可用结构"。
3. **三种推理触发器**：Zero-shot（简单任务）、Few-shot（需要模仿）、Chain-of-Thought（需要推理）。
4. **系统提示词 vs 用户提示词**：系统提示词定义"游戏规则"，用户提示词是"游戏内的操作"。
5. **Prompt版本管理**：Prompt需要像代码一样被管理——版本控制、A/B测试、灰度发布。
6. **客户工单案例**：从62%到88%的准确率提升，关键是加了Few-shot示例和约束条件。

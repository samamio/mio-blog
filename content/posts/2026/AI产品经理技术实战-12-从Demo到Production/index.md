# 第12章 企业级AI系统架构——从Demo到Production

## 本章目标

第十一章你学会了怎么写AI产品的PRD。但PRD写得再好，如果系统架构撑不住，上线就会出问题。

一个在本地跑Demo很好的AI功能，放到生产环境可能会遇到：

- 并发量上去了，响应时间从1秒变成10秒
- 模型API挂了，整个产品不能用
- Token费用一个月烧了100万
- 模型输出了不应该输出的内容

**企业级AI系统架构解决的就是这些问题。**

学习完本章后，你将能够：

1. **理解生产环境中AI系统的完整架构**——从用户请求到模型响应到结果返回的全链路
2. **设计模型路由策略**——不同场景用不同模型，平衡效果和成本
3. **设计缓存和降级策略**——系统挂了怎么办
4. **建立监控和告警体系**——出了问题第一时间知道
5. **优化AI系统的成本**——Token费用怎么省
6. **设计灰度发布方案**——新功能怎么安全地上线

## 一、背景：Demo和Production的区别

### 1.1 Demo环境 vs Production环境

| 维度 | Demo环境 | Production环境 |
|------|---------|---------------|
| 用户量 | 1-5人 | 1万-100万人 |
| 并发量 | 1-2个请求 | 1000+个请求/秒 |
| 数据量 | 10条测试数据 | 1000万+条真实数据 |
| 网络 | 本地局域网 | 公网 |
| 监控 | 没有 | 完整监控体系 |
| 容错 | 挂了重启就行 | 需要高可用 |
| 成本 | 不考虑 | 严格控制 |

### 1.2 从Demo到Production的常见坑

**坑一：并发量爆炸**

Demo时每秒1个请求，Production时每秒1000个请求。模型API的限流策略没设置，直接被ban。

**坑二：Token费用失控**

Demo时每天10条请求，每条1000 Token。Production时每天10万条请求，每条5000 Token。月费用从100块变成100万。

**坑三：模型响应超时**

Demo时模型300ms响应。Production时因为网络延迟、排队、模型负载，响应时间变成5秒。

**坑四：数据泄露**

Demo时数据都是假的。Production时数据是真的，但没有做好权限控制和加密。

**坑五：模型输出不可控**

Demo时Prompt写得很简单。Production时用户输入各种奇怪的内容，Prompt被绕过，模型输出了不应该输出的内容。

### 2.2 关键组件详解

#### 组件一：API Gateway（接入层）

API Gateway是整个系统的入口，负责：

1. **认证**：验证用户身份（OAuth 2.0、JWT Token）
2. **限流**：防止并发量过高（每秒最多1000个请求）
3. **路由**：根据请求类型路由到不同的服务
4. **日志**：记录所有请求的日志

**产品经理需要了解：**

- 限流策略直接影响用户体验——如果限流太严，用户请求会被拒绝
- 需要在PRD中定义限流策略：每个用户每秒最多N个请求

#### 组件二：模型路由（Model Router）

模型路由根据请求的类型，选择最合适的模型：

```
请求进来 → 判断类型 → 选择模型
                    ├── 简单分类 → Qwen-7B（便宜、快）
                    ├── 复杂推理 → Claude 3 Opus（贵、慢但好）
                    ├── 多语言 → GPT-4o（均衡）
                    └── 中文优先 → Qwen-72B（自建）
```

**产品经理需要了解：**

- 模型路由直接影响成本和效果
- 需要在PRD中定义"什么场景用什么模型"
- 可以设计"模型效果 vs 成本"的权衡策略

#### 组件三：缓存层（Cache）

缓存是降低延迟和成本的关键：

```
用户提问："怎么重置密码？"
    │
    ▼
缓存查询：这个问题之前有人问过吗？
    │
    ├── 有缓存 → 直接返回缓存结果（0ms）
    └── 无缓存 → 调用模型（2000ms）
              │
              ▼
         缓存结果（TTL=24小时）
```

**缓存策略：**

| 缓存类型 | 缓存内容 | TTL |
|---------|---------|-----|
| 查询缓存 | 相同问题的相同回答 | 24小时 |
| 结果缓存 | 工具调用结果（如订单信息） | 5分钟 |
| 模板缓存 | Prompt模板 | 永久（变更时刷新） |

**产品经理需要了解：**

- 缓存命中率直接影响用户体验和成本
- 需要在PRD中定义"哪些结果可以缓存、缓存多久"
- 缓存过期策略影响数据的时效性

#### 组件四：降级服务（Fallback）

当AI服务不可用时，降级策略保证产品还能用：

```
AI服务正常 → 正常回复
AI服务超时 → 返回缓存结果
AI服务挂掉 → 转人工客服
AI服务效果差 → 降级为规则引擎
```

**降级策略：**

| 故障类型 | 降级策略 |
|---------|---------|
| 模型API超时 | 返回缓存结果 |
| 模型API挂掉 | 转人工客服 |
| 模型效果差 | 降级为关键词匹配 |
| 向量数据库挂掉 | 降级为关键词搜索 |
| 全部挂掉 | 显示"系统维护中" |

**产品经理需要了解：**

- 降级策略是用户体验的重要组成部分
- 需要在PRD中定义"什么情况下触发降级"和"降级后用户看到什么"
- 降级不等于失败——好的降级策略让用户几乎感知不到问题

## 三、企业应用：AI系统的监控和告警

### 3.1 监控指标体系

一个企业级AI系统需要监控以下指标：

**性能指标：**

| 指标 | 说明 | 目标 |
|------|------|------|
| P50延迟 | 50%的请求在此时间内完成 | <1秒 |
| P95延迟 | 95%的请求在此时间内完成 | <3秒 |
| P99延迟 | 99%的请求在此时间内完成 | <5秒 |
| 吞吐量 | 每秒处理的请求数 | >1000 req/s |

**效果指标：**

| 指标 | 说明 | 目标 |
|------|------|------|
| 准确率 | 模型回答正确的比例 | >85% |
| 用户满意度 | 用户标记"有帮助"的比例 | >75% |
| 自动解决率 | AI自动解决的比例 | >60% |
| 转人工率 | 需要转人工的比例 | <20% |

**成本指标：**

| 指标 | 说明 | 目标 |
|------|------|------|
| 单次请求成本 | 每次请求的Token费用 | <$0.01 |
| 日/月总成本 | 每日/每月的总费用 | 预算内 |
| Token利用率 | 有效Token / 总Token | >80% |

**系统指标：**

| 指标 | 说明 | 目标 |
|------|------|------|
| 服务可用性 | 系统正常运行的时间比例 | >99.9% |
| 缓存命中率 | 缓存命中的比例 | >60% |
| 降级触发率 | 触发降级的比例 | <1% |
| 错误率 | 请求失败的比例 | <0.1% |

### 3.2 告警策略

```
P0级告警（立即响应）：
- 服务不可用（错误率>10%）
- 模型API全部挂掉
- 数据泄露

P1级告警（30分钟内响应）：
- 准确率下降到70%以下
- 响应时间P95>10秒
- 单日Token费用超过预算的80%

P2级告警（当天处理）：
- 缓存命中率下降到30%以下
- 转人工率上升到30%以上
- 用户满意度下降到60%以下
```

## 四、产品经理需要掌握什么？

### 4.1 架构设计的产品思维

作为产品经理，你不需要设计技术架构，但你需要理解架构对产品的影响：

**影响一：用户体验**

- 缓存命中率高 → 响应快 → 用户体验好
- 降级策略合理 → 服务不断 → 用户体验好
- 模型路由合理 → 效果好 → 用户体验好

**影响二：成本**

- 模型选择 → 成本差异可达10倍
- 缓存策略 → 缓存命中率高可降低50%以上的Token费用
- 限流策略 → 限流太严会拒掉正常用户

**影响三：可靠性**

- 多模型冗余 → 一个模型挂了可以用另一个
- 多地域部署 → 一个地域挂了可以切换到另一个
- 灰度发布 → 新版本有问题可以快速回滚

### 4.2 架构相关的PRD写法

在PRD中，需要包含以下架构相关内容：

**模块一：性能要求**

```
- 响应时间：P95 < 3秒
- 并发能力：支持1000 req/s
- 可用性：99.9%
```

**模块二：降级策略**

```
- 模型API超时 → 返回缓存结果
- 模型API挂掉 → 转人工客服
- 向量数据库挂掉 → 降级为关键词搜索
```

**模块三：缓存策略**

```
- 查询缓存：相同问题24小时内直接返回缓存
- 结果缓存：工具调用结果缓存5分钟
- 缓存命中率目标：>60%
```

**模块四：监控要求**

```
- 实时监控：请求数、响应时间、准确率、成本
- 告警阈值：准确率<80%告警、成本超预算80%告警
- 日报/周报：效果趋势分析
```

### 4.3 跟工程师沟通的常用话术

**讨论架构时：**

> "如果模型API挂了，用户的体验是什么？是看到错误信息还是自动转人工？"

**讨论成本时：**

> "这个功能每个月大概消耗多少Token？如果用户量翻10倍，成本会怎样变化？"

**讨论性能时：**

> "P95响应时间能不能做到3秒以内？如果做不到的话，有没有降级方案？"

## 五、研发如何实现：成本优化

### 5.1 Token费用优化策略

**策略一：模型选择优化**

```
简单任务（分类、提取）→ 7B模型（$0.001/千Token）
中等任务（摘要、翻译）→ 70B模型（$0.01/千Token）
复杂任务（推理、创作）→ 大模型（$0.1/千Token）
```

**策略二：缓存优化**

```
假设每天10000个请求：
- 无缓存：10000 × 1000 Token = 10,000,000 Token
- 缓存命中率50%：5000 × 1000 Token = 5,000,000 Token
- 节省：50%
```

**策略三：Prompt优化**

```
- 精简Prompt：从2000字精简到500字，节省75%的输入Token
- 减少示例：Few-shot从10个示例减少到3个，节省60%的Token
- 按需检索：RAG只检索最相关的3个chunk，而不是10个
```

**策略四：批处理**

```
- 单个请求：每次调用一次模型
- 批处理：每10个请求合并成一次调用
- 节省：批量调用的单价通常更低
```

### 5.2 成本估算模板

```
月度成本估算：

日均请求数：10,000
每请求平均Token：
  - 输入：500 Token
  - 输出：200 Token
  - 合计：700 Token

月输入Token：10,000 × 70 × 500 = 350,000,000
月输出Token：10,000 × 70 × 200 = 140,000,000

成本（GPT-4o）：
  - 输入：350,000,000 / 1000 × $0.01 = $3,500
  - 输出：140,000,000 / 1000 × $0.03 = $4,200
  - 合计：$7,700/月

成本优化后（混合模型+缓存）：
  - 简单任务用7B模型：节省80%
  - 缓存命中率50%：节省50%
  - 合计：$7,700 × 20% × 50% = $770/月
```

## 七、真实案例：某AI客服系统的生产上线

### 7.1 背景

某公司的AI客服系统在Demo环境下效果很好（准确率92%），但上线后效果骤降到65%。

### 7.2 问题分析

**问题一：并发量爆炸**

Demo时每秒1个请求，上线后每秒1000个请求。模型API的限流策略没设置，大量请求被拒绝。

**问题二：用户输入不可控**

Demo时用的人工构造的测试数据，上线后是真实用户的各种奇怪输入。Prompt没考虑到这些边界情况。

**问题三：缓存策略缺失**

没有缓存，每个请求都调用模型，成本高且响应慢。

**问题四：没有降级策略**

模型API超时后，没有降级方案，用户直接看到错误信息。

### 7.3 解决方案

```
1. 设置限流策略：每个IP每秒最多10个请求
2. 增加输入清洗：过滤异常输入
3. 添加缓存层：相同问题24小时内直接返回缓存
4. 设计降级策略：模型超时返回缓存结果，模型挂掉转人工
5. 增加监控：实时跟踪准确率、响应时间、成本
6. 灰度发布：先1%用户，效果正常后逐步扩大到100%
```

### 7.4 效果

上线一个月后：

- 准确率从65%恢复到85%
- 平均响应时间从8秒降到2秒
- 月度Token费用从$5000降到$800（缓存+模型路由优化）
- 用户满意度从60%提升到82%

## 八、常见误区

### 误区一："Demo效果好，上线就一定好"

**真相**：Demo环境和生产环境差异巨大。上线前需要做充分的压测和灰度。

### 误区二："成本越低越好"

**真相**：成本优化不能以牺牲用户体验为代价。如果缓存命中率太高导致返回过时信息，反而降低满意度。

### 误区三："监控越多越好"

**真相**：监控指标太多会导致"告警疲劳"——所有告警都一样重要，等于没有告警。需要分级管理。

### 误区四："降级就是失败"

**真相**：好的降级策略是用户体验的保障。用户感知不到降级，比看到错误信息要好得多。

### 误区五："架构设计是研发的事"

**真相**：架构直接影响用户体验、成本和可靠性。产品经理需要深度参与架构设计决策。

### 误区六："一次性设计好，以后不用改"

**真相**：架构需要随业务增长而演进。初期的简单架构在用户量增长后需要重构。

## 九、面试题

### 初级

**Q1：Demo环境和生产环境有什么区别？**

**参考答案：**

主要区别：

1. 用户量：Demo是几个人，生产是成千上万
2. 数据质量：Demo是人工构造的干净数据，生产是真实用户的脏数据
3. 并发量：Demo是1-2个请求，生产是几百上千个请求/秒
4. 监控：Demo没有监控，生产需要完整的监控体系
5. 容错：Demo挂了重启就行，生产需要高可用

---

### 中级

**Q2：如何降低AI产品的Token成本？**

**参考答案：**

四个策略：

1. **模型路由**：简单任务用小模型，复杂任务用大模型
2. **缓存**：相同问题直接返回缓存结果，不调用模型
3. **Prompt优化**：精简Prompt，减少不必要的Token消耗
4. **批处理**：合并相似请求，减少调用次数

---

### 高级

**Q3：一个AI产品上线后准确率从92%降到65%，你怎么排查和解决？**

**参考答案：**

**排查步骤：**

1. **检查数据质量**：对比Demo数据和线上数据的差异。线上数据可能更脏、更多样。
2. **分析Bad Case**：分类统计错误类型。是模型问题、Prompt问题、还是数据问题？
3. **检查缓存**：缓存的数据是否过期了？缓存命中率是否正常？
4. **检查模型**：模型版本有没有更新？更新后效果有没有变化？
5. **检查并发**：高并发时模型响应时间是否变长？超时是否影响了效果？

**解决方案：**

1. 增加输入清洗和校验
2. 优化Prompt，增加边界情况的处理
3. 更新缓存策略
4. 如果模型版本更新了，回滚到旧版本
5. 增加限流和降级策略

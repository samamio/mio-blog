# 第11章 AI产品的PRD怎么写——与传统PRD的五大差异


---

## 一、背景：为什么AI产品需要新的PRD？

### 1.1 传统PRD的问题

传统PRD是为"确定性产品"设计的。它的核心假设是：

> 如果需求描述清楚了，研发就能实现，实现后功能就一定对。

这个假设在确定性产品中成立，但在AI产品中不成立。

**传统PRD的典型写法：**

```
功能：智能搜索
- 用户输入关键词后，系统搜索匹配的商品
- 按相关度排序，返回前10条结果
- 如果结果为空，显示"没有找到相关商品"
```

**问题：** 如果"相关度"的定义是什么？"匹配"是关键词匹配还是语义匹配？"相关度排序"的算法是什么？传统PRD不说清楚，因为传统PRD假设研发会自己决定。

但在AI产品中，"相关度"的定义直接决定产品效果。如果定义错了，用户搜"苹果手机"可能返回的是"水果"而不是"iPhone"。

### 1.2 AI产品PRD的核心转变

AI产品PRD的核心转变可以概括为一句话：

> **从"描述功能怎么实现"到"定义能力边界和效果标准"。**

传统PRD：告诉研发"做什么"和"怎么做"。
AI PRD：告诉研发"做什么"和"做到什么程度算好"。

## 二、核心原理：AI产品PRD的五大差异

### 差异一：确定性 vs 概率性

**传统PRD：**

```
用户点击"搜索"按钮后，系统返回包含关键词的商品列表。
```

确定性描述——同样的输入，同样的输出。

**AI PRD：**

```
用户输入自然语言问题后，系统在2秒内返回AI回复。回复的准确率目标为85%。当回复置信度低于70%时，显示"我正在学习中，您的回答对我们很重要"并转人工客服。
```

概率性描述——同样的输入，可能不同的输出。需要定义"足够好"的标准和兜底策略。

### 差异二：功能描述 vs 能力描述

**传统PRD：**

```
功能：用户注册
- 输入：用户名、密码、邮箱
- 输出：注册成功/失败
- 验证：用户名不能重复，密码长度≥8位
```

**AI PRD：**

```
能力：智能客服
- 能做什么：理解用户问题，基于知识库生成回复
- 不能做什么：不能处理退款（需人工）、不能处理投诉（需人工）
- 效果指标：准确率≥85%，响应时间<3秒
```

### 差异三：验收标准 vs 效果指标

**传统PRD：**

```
验收标准：
- [ ] 用户输入关键词能搜索到商品
- [ ] 搜索结果为空时显示提示
- [ ] 按相关度排序
```

二元判断——功能要么对了，要么错了。

**AI PRD：**

```
效果指标：
- [ ] 总体准确率≥85%
- [ ] 每类问题的召回率≥80%
- [ ] 95%的请求在3秒内响应
- [ ] 用户满意度≥75%
```

连续评估——效果有好有坏，需要量化标准。

### 差异四：一次性交付 vs 持续迭代

**传统PRD：**

```
上线 = 交付完成
```

**AI PRD：**

```
上线 = 开始持续优化
- 每周分析Bad Case
- 每月评估效果指标
- 每季度审查Prompt和知识库
- 持续收集用户反馈
```

AI产品上线不是终点，而是起点。

### 差异五：功能版本 vs 模型/Prompt版本

**传统PRD：**

```
版本管理：
- v1.0：搜索功能上线
- v1.1：增加筛选条件
- v1.2：增加排序方式
```

**AI PRD：**

```
版本管理：
- 功能版本：v1.0（智能客服上线）
- Prompt版本：v3.2（工单分类Prompt）
- 模型版本：GPT-4 Turbo（2024-04-09）
- 知识库版本：2024-06-01（每周更新）
```

AI产品的版本管理是多维的——功能、Prompt、模型、知识库各有自己的版本。

### 3.2 每个模块的详细写法

**模块一：能力描述**

```
能做什么：
- 理解用户的自然语言问题
- 基于知识库生成回复
- 调用工具查询订单和物流信息
- 自动分类工单

不能做什么：
- 处理退款（需人工）
- 处理投诉（需人工）
- 理解纯图片工单
- 处理不在知识库中的问题

能力边界：
- 支持中文和英文
- 不支持其他语言
- 知识库更新时间：每周更新
```

**模块二：效果指标**

```
总体指标：
- 回复准确率：≥85%
- 工单分类准确率：≥90%
- 响应时间：P95 < 3秒
- 用户满意度：≥75%

分场景指标：
- 简单问题（FAQ类）：准确率≥90%
- 中等问题（需要查询）：准确率≥80%
- 复杂问题（需要推理）：准确率≥70%
```

**模块三：Bad Case处理**

```
分级策略：
- 置信度≥85%：自动回复
- 70%≤置信度<85%：标记"待确认"，人工审核后发送
- 置信度<70%：转人工客服

兜底方案：
- 模型超时（>5秒）：显示"正在处理中，请稍后"
- 知识库为空：转人工客服
- 工具调用失败：降级为纯模型回答

反馈机制：
- 用户点击"没有帮助"：自动归档Bad Case
- 客服修改AI建议：记录修改前后的差异
- 每周生成Bad Case报告
```

## 四、产品经理需要掌握什么？

### 4.1 AI产品PRD的写作技巧

**技巧一：用数据说话**

不要写："希望模型回答得准确一些"
要写："回复准确率≥85%"

不要写："响应速度要快"
要写："P95响应时间<3秒"

**技巧二：定义"不好"的情况**

传统PRD定义"功能不对"的情况。
AI PRD定义"效果不好"的情况。

**技巧三：给出"足够好"的标准**

不是追求100%准确，而是追求"足够好"。

**技巧四：设计兜底方案**

每个AI功能都要有兜底方案——当AI做不好的时候，人来做。

### 4.2 效果指标的设计方法

**步骤一：定义业务指标**

```
业务目标：降低客服成本
业务指标：人工客服工作量减少50%
```

**步骤二：映射到AI效果指标**

```
人工客服工作量减少50% ← 自动解决率达到60%
自动解决率达到60% ← 回复准确率达到85%
回复准确率达到85% ← 模型置信度阈值设为70%
```

**步骤三：设定阶段性目标**

```
第一期：自动解决率40%，准确率75%
第二期：自动解决率50%，准确率80%
第三期：自动解决率60%，准确率85%
```

### 4.3 Bad Case库的建设方法

**收集：**

```
- 用户标记"没有帮助"
- 客服修改AI建议
- 自动检测低置信度回复
- 定期抽样人工审核
```

**分类：**

```
- 知识库缺失：知识库中没有相关内容
- 检索不准：检索到了但不相关
- Prompt问题：模型理解错了意图
- 模型能力：模型本身做不到
- 产品流程：兜底策略不合理
```

**优先级排序：**

```
- P0：影响大量用户的高频问题
- P1：影响少量用户的高频问题
- P2：低频但严重的问题（如安全问题）
- P3：低频且不严重的问题
```

**回归测试：**

```
- 每周：检查本周的Bad Case是否在历史中出现过
- 每月：用标准化测试集重新评估效果
- 每次Prompt/知识库变更后：回归测试
```

## 五、客户工单系统的最终PRD

### 5.1 概述

```
产品名称：智能客服工单系统
产品定位：基于AI的自动化客服工单处理系统
目标用户：SaaS产品的终端用户
业务目标：
- 人工客服工作量减少50%
- 用户满意度从72%提升到85%
- 平均响应时间从4小时降到30分钟以内
```

### 5.2 能力描述

```
能做什么：
- 自动分类工单（技术故障/功能建议/账号问题/其他）
- 基于知识库生成回复建议
- 查询用户的历史工单
- 转接人工客服

不能做什么：
- 处理退款（需人工）
- 处理投诉（需人工）
- 理解纯图片工单
- 处理知识库中没有的问题

能力边界：
- 支持中文和英文
- 知识库每周更新
- 模型：Qwen-72B（私有化部署）
```

### 5.3 效果指标

```
总体指标：
- 工单分类准确率：≥90%
- 回复建议采纳率：≥70%
- 响应时间：P95 < 5秒
- 用户满意度：≥80%

分场景指标：
- 简单问题（FAQ类）：准确率≥95%
- 中等问题（需要查询）：准确率≥85%
- 复杂问题（需要推理）：准确率≥70%
```

### 5.4 人机协同设计

```
自动执行：
- 工单分类
- 知识库检索
- 回复建议生成

人工确认：
- 退款操作
- 投诉处理
- 置信度<70%的回复

必须转人工：
- 用户明确要求人工
- 涉及隐私信息
- 情绪负面（愤怒、威胁）
```

### 5.5 版本管理

```
功能版本：v1.0（2024-06-01上线）
Prompt版本：
- 工单分类Prompt：v3.2
- 回复建议Prompt：v2.1
模型版本：Qwen-72B（2024-05-01部署）
知识库版本：2024-06-01（每周更新）
```

### 5.6 数据反馈机制

```
用户反馈：
- 每条回复后有"有帮助/没有帮助"按钮
- 点击"没有帮助"时弹出简短原因选择

客服反馈：
- 客服可以修改AI建议的回复
- 修改时记录修改前后的差异

效果监控：
- 每日：自动解决率、用户满意度
- 每周：Bad Case报告、Prompt效果分析
- 每月：效果指标趋势、知识库质量评估
```

## 七、真实案例：某电商平台的AI推荐PRD

### 7.1 背景

某电商平台希望用AI重构推荐系统。原来的推荐系统是协同过滤（"买了A的人也买了B"），效果一般。

### 7.2 AI推荐PRD

```
能力描述：
- 能做什么：理解用户的自然语言描述，推荐相关商品
- 不能做什么：不能推荐库存为0的商品，不能推荐用户所在地区不可达的商品
- 效果指标：推荐点击率≥15%（原系统8%），转化率≥3%（原系统1.5%）

Bad Case处理：
- 推荐了库存为0的商品 → 自动替换为相似有货商品
- 用户点击"不感兴趣" → 记录偏好，下次减少同类推荐

人机协同：
- 自动推荐Top 5
- 第6-10位由人工编辑推荐（保证多样性）

版本管理：
- 推荐模型：v2.1
- Prompt：v1.0
- 知识库：实时（商品库存）
```

### 7.3 效果

上线三个月后：

- 推荐点击率从8%提升到18%
- 转化率从1.5%提升到3.5%
- GMV提升12%

## 八、常见误区

### 误区一："AI产品的PRD跟传统PRD差不多"

**真相**：差异很大。AI产品PRD需要定义效果指标、Bad Case处理、人机协同，这些在传统PRD中是不需要的。

### 误区二："效果指标设得越高越好"

**真相**：效果指标需要和成本平衡。追求99%的准确率可能需要10倍的成本，但业务收益可能只增加1%。

### 误区三："Bad Case库建一次就够了"

**真相**：Bad Case库需要持续更新。新的Bad Case每天都在产生，旧的可能已经修复了。

### 误区四："PRD写完了就可以交付了"

**真相**：AI产品的PRD是活的文档。每次模型更新、Prompt迭代、知识库变更，都需要更新PRD。

### 误区五："人机协同就是加一个人工审核步骤"

**真相**：人机协同需要精心设计——哪些自动、哪些确认、哪些转人工，每个决策都有产品影响。

### 误区六："版本管理不重要"

**真相**：AI产品的版本管理比传统产品复杂得多——功能、Prompt、模型、知识库各有版本。不管理好版本，出问题时无法回滚。

### 误区七："数据反馈机制是研发的事"

**真相**：数据反馈机制是产品设计的核心部分。产品经理需要定义收集什么数据、怎么分析、怎么迭代。

## 九、面试题

### 初级

**Q1：AI产品的PRD和传统PRD有什么区别？**

**参考答案：**

五大区别：

1. 确定性vs概率性：传统PRD描述确定性行为，AI PRD描述概率性行为
2. 功能描述vs能力描述：传统PRD描述功能怎么做，AI PRD描述能力边界
3. 验收标准vs效果指标：传统PRD看功能对错，AI PRD看效果好坏
4. 一次性交付vs持续迭代：传统PRD上线即完成，AI PRD上线才开始
5. 单一版本vs多维版本：传统PRD只有一个功能版本，AI PRD有功能、Prompt、模型、知识库多个版本

---

**Q2：如何定义AI产品的"做好了"？**

**参考答案：**

从三个维度定义：

1. 技术指标：准确率、召回率、响应时间
2. 用户体验指标：用户满意度、任务完成率
3. 业务指标：成本节约、收入提升、效率改善

三个维度需要同时看，不能只看一个。

---

### 中级

**Q3：你负责一个AI产品，上线后用户满意度只有60%。你如何分析和改进？**

**参考答案：**

**第一步：数据收集**

- 收集过去一个月的用户反馈数据
- 标记"有帮助"和"没有帮助"的案例各100条

**第二步：Bad Case分析**

- 分类Bad Case：知识库缺失30%、检索不准25%、Prompt问题20%、模型能力15%、产品流程10%
- 找出Top 3问题

**第三步：制定改进计划**

- P0：补充知识库缺失内容（30%的Bad Case）
- P1：优化检索策略（25%的Bad Case）
- P2：迭代Prompt设计（20%的Bad Case）

**第四步：实施和验证**

- 每两周迭代一次
- 每次迭代后重新评估用户满意度

### 高级

**Q4：你负责一个AI客服产品，业务方要求准确率做到99%。你怎么办？**

**参考答案：**

首先，我会和业务方沟通，了解99%这个数字是怎么来的。

然后，我会给出务实的方案：

**方案A：纯AI做到99%**
- 技术上几乎不可能。当前最好的AI客服准确率也就90-95%
- 成本极高，需要大量人工标注数据

**方案B：AI+人工做到99%**
- AI处理80%的简单问题（准确率90%+）
- 复杂问题转人工（准确率100%）
- 整体准确率 = 80% × 90% + 20% × 100% = 92%
- 继续优化AI，直到整体准确率达到99%

**我的建议**：跟业务方沟通，目标不是"AI准确率99%"，而是"用户满意度99%"。后者可以通过AI+人工的组合实现，前者几乎不可能。

**Q5：请设计一个AI产品的版本管理方案。**

**参考答案：**

**版本维度：**

1. 功能版本：整体产品的版本号（v1.0, v1.1, v2.0）
2. Prompt版本：每个场景的Prompt版本号（分类Prompt v3.2, 回复Prompt v2.1）
3. 模型版本：使用的模型版本（Qwen-72B-20240501, GPT-4-Turbo-20240409）
4. 知识库版本：知识库的更新日期（2024-06-01）

**管理方式：**

- 所有版本信息存储在版本管理系统中
- 每次变更记录：变更内容、变更原因、变更人、变更时间
- 支持回滚到任意历史版本
- 每次上线前记录"版本快照"

**灰度策略：**

- 新功能：先对10%用户灰度
- 新Prompt：先对5%用户灰度
- 新模型：先对1%用户灰度
- 灰度期间密切监控效果指标
```

## 十、本章总结

本章我们系统学习了AI产品PRD的写法和最佳实践：

1. **五大差异**：确定性vs概率性、功能vs能力、验收vs效果、交付vs迭代、单一版本vs多维版本

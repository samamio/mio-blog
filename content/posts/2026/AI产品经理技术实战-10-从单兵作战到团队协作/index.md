# 第10章 多Agent系统——从单兵作战到团队协作

## 本章目标

第九到十章我们已经学了MCP标准化连接协议。但即使有了标准化协议，**一个Agent能做的事情仍然是有限的。**

想象一下：

> 一个客服Agent需要处理一个复杂的工单：
> - 需要先理解用户的问题
> - 查询用户的历史订单
> - 检索知识库找到解决方案
> - 判断是否需要退款
> - 如果需要退款，计算退款金额
> - 生成回复并发送
> - 最后质检回复质量

一个Agent同时做这么多事，会出现什么问题？

1. **Token超限**：每一步的中间结果都占Token，5-6步之后上下文窗口就满了
2. **注意力分散**：Agent同时关注太多任务，每件事都做不好
3. **单点故障**：某一步失败了，整个流程崩溃
4. **调试困难**：一个Agent有5个步骤，出问题时不知道是哪一步的问题

**解决方案：把一个大Agent拆成多个小Agent，各司其职，团队协作。**

这就是多Agent系统的价值。

学习完本章后，你将能够：

1. **理解为什么单Agent走不远**——复杂度增长带来的能力稀释
2. **掌握多Agent的三种组织模式**——Master-Worker、Peer-to-Peer、Swarm
3. **设计Agent角色**——产品经理式的能力画像与接口定义
4. **设计Agent间通信协议**——共享状态、消息传递、同步机制
5. **处理冲突和一致性**——当两个Agent给出不同结论怎么办
6. **将客户工单系统升级到多Agent架构**

## 一、背景：为什么需要多Agent？

### 1.1 单Agent的能力极限

一个Agent能同时做的事情是有上限的。这个上限由以下因素决定：

**因素一：上下文窗口**

即使是最强的模型（200K Token上下文），也不够用。

举例：

```
每步消耗的Token：
- 系统提示词：2000
- 用户消息：500
- 上一步的Thought：1000
- 上一步的Observation：2000
- 工具描述：5000
- 对话历史（每轮）：3000

5轮对话后：
2000 + 500 + (1000+2000+5000+3000) × 5 = 42,500 Token

10轮对话后：
2000 + 500 + (1000+2000+5000+3000) × 10 = 72,500 Token

15轮对话后：
2000 + 500 + (1000+2000+5000+3000) × 15 = 102,500 Token
```

15轮之后，128K Token的上下文窗口就不够了。

**因素二：注意力分散**

一个Agent同时处理5个任务时，每个任务的注意力只有20%。结果是每个任务都做得不好。

**因素三：单点故障**

一个Agent有5个步骤，任何一个步骤出错，整个流程崩溃。

### 1.2 多Agent的核心价值

**价值一：专业化**

每个Agent专注于一个任务，做得更深更精。

**价值二：可扩展性**

新增能力时，加一个Agent就行，不需要改现有的。

**价值三：容错性**

一个Agent失败了，其他Agent还能继续工作。

**价值四：可调试性**

出问题时可以定位到是哪个Agent的问题。

### 1.3 多Agent vs 单体Agent：成本对比

| 维度 | 单体Agent | 多Agent |
|------|---------|--------|
| Token消耗 | 高（所有上下文在一个对话中） | 低（每个Agent独立对话） |
| 模型选择 | 需要大模型（上下文大） | 可以用小模型（每个Agent任务简单） |
| 并行度 | 串行（一步一步来） | 可以并行（多个Agent同时工作） |
| 调试难度 | 高（一个Agent里找问题） | 低（定位到具体Agent） |
| 开发复杂度 | 低 | 高 |
| 维护成本 | 低 | 高 |

**结论**：多Agent的Token成本通常更低（因为每个Agent可以用小模型），但开发和维护成本更高。

## 二、核心原理：多Agent的组织模式

### 2.1 模式一：Master-Worker（主从模式）

**类比：公司组织架构**

```
          CEO (Master)
         /    |    \
        /     |     \
    技术总监  市场总监  财务总监
    (Worker) (Worker) (Worker)
```

Master负责任务分解和协调，Worker负责具体执行。

**多Agent实现：**

```
Planner Agent（Master）
    ├── Query Agent（Worker）—— 查询工单
    ├── Reply Agent（Worker）—— 生成回复
    └── Escalate Agent（Worker）—— 转接人工
```

**Planner的工作流程：**

```
1. 接收用户请求
2. 分解任务：需要查询工单 + 生成回复
3. 分配给对应的Worker Agent
4. 收集Worker的结果
5. 整合结果，生成最终回复
```

**适用场景：**

- 任务可以明确分解成子任务
- 需要一个"大脑"来协调
- 子任务之间没有复杂的依赖关系

### 2.2 模式二：Peer-to-Peer（对等模式）

**类比：项目组**

```
成员A ↔ 成员B ↔ 成员C
  ↕       ↕       ↕
互相沟通，互相协作
```

所有Agent地位平等，可以自由通信和协作。

**多Agent实现：**

```
接待Agent ↔ 查询Agent ↔ 决策Agent ↔ 质检Agent
   ↕          ↕          ↕          ↕
信息自由流动，互相传递
```

**工作流程：**

```
用户消息 → 接待Agent
  → 接待Agent把消息转发给查询Agent
  → 查询Agent查询工单后返回给接待Agent
  → 接待Agent把结果转发给决策Agent
  → 决策Agent决定回复内容后返回给接待Agent
  → 接待Agent把回复转发给质检Agent
  → 质检Agent检查回复质量
  → 质量通过 → 发送给用户
```

**适用场景：**

- 任务之间有较强的依赖关系
- Agent之间需要频繁交换信息
- 没有单一的"领导者"

### 2.3 模式三：Swarm（蜂群模式）

**类比：蜜蜂群落**

```
蜂后（Queen）
  │
  ├── 工蜂A：采集花蜜
  ├── 工蜂B：采集花蜜
  ├── 工蜂C：采集花蜜
  └── 工蜂D：采集花蜜

每个工蜂独立工作，发现花蜜后返回蜂巢
蜂后不需要指挥，工蜂之间也不需要协调
```

**多Agent实现：**

```
每个Agent独立处理一个工单，互不干扰
```

**工作流程：**

```
用户消息池 → Agent池 → 回复结果池 → 用户

每个Agent从消息池中领取一个消息，独立处理，完成后放回结果池。
```

**适用场景：**

- 大量相似的独立任务
- 任务之间没有依赖关系
- 需要高吞吐量和可扩展性

### 2.4 三种模式的对比

| 维度 | Master-Worker | Peer-to-Peer | Swarm |
|------|--------------|--------------|-------|
| 复杂度 | 中 | 高 | 低 |
| 灵活性 | 中 | 高 | 低 |
| 可扩展性 | 中 | 中 | 高 |
| 调试难度 | 中 | 高 | 低 |
| 适用场景 | 有明确任务分解 | 复杂协作 | 大量独立任务 |

## 三、企业应用：多Agent的设计方法

### 3.1 Agent角色设计

设计多Agent系统的第一步是**定义每个Agent的角色**。

**角色设计的三个问题：**

1. **这个Agent做什么？**（职责）
2. **这个Agent需要什么能力？**（技能）
3. **这个Agent和其他Agent怎么协作？**（接口）

**角色设计示例：客服多Agent系统**

| Agent | 职责 | 技能 | 输入 | 输出 |
|-------|------|------|------|------|
| 接待Agent | 接收用户消息，初步理解 | 意图识别、情感分析 | 用户消息 | 结构化请求 |
| 查询Agent | 查询工单和订单信息 | Function Calling、RAG | 结构化请求 | 查询结果 |
| 决策Agent | 决定如何处理 | 推理、判断 | 查询结果 | 处理方案 |
| 回复Agent | 生成用户回复 | 文案生成、语气控制 | 处理方案 | 回复草稿 |
| 质检Agent | 检查回复质量 | 质量评估 | 回复草稿 | 通过/不通过 |

### 3.2 Agent间通信协议

多Agent之间需要通信。通信的方式有三种：

**方式一：共享状态（Shared State）**

所有Agent共享同一个状态对象。每个Agent读取和更新这个状态。

```
State = {
    user_message: "我的订单到哪了",
    ticket_info: {...},
    reply: "",
    quality_score: 0.0
}

接待Agent → 写入 user_message
查询Agent → 读取 user_message，写入 ticket_info
决策Agent → 读取 ticket_info，写入 decision
回复Agent → 读取 decision，写入 reply
质检Agent → 读取 reply，更新 quality_score
```

**方式二：消息传递（Message Passing）**

Agent之间通过消息通信。一个Agent发送消息给另一个Agent，另一个Agent处理后回复。

```
接待Agent → 发送消息 → 查询Agent
查询Agent → 返回结果 → 接待Agent
接待Agent → 发送消息 → 决策Agent
决策Agent → 返回方案 → 接待Agent
...
```

**方式三：混合模式**

简单任务用消息传递，复杂协作用共享状态。

### 3.3 冲突解决

当多个Agent给出不同结论时，怎么解决？

**策略一：优先级**

给每个Agent设一个优先级。优先级高的Agent的结论为准。

```
决策Agent (优先级1) → 最终决策
质检Agent (优先级2) → 否决决策Agent的方案
```

**策略二：投票**

多个Agent独立判断，多数决。

```
3个质检Agent独立评分：
Agent A: 85分
Agent B: 82分
Agent C: 78分
平均分: 81.7 → 通过
```

**策略三：人工仲裁**

Agent之间无法达成一致时，转人工。

### 3.4 多Agent的成本优化

多Agent意味着多次模型调用，成本可能很高。优化策略：

**策略一：用小模型做简单任务**

```
意图分类 → 7B参数模型（便宜）
复杂推理 → 70B参数模型（贵）
```

**策略二：并行执行**

```
查询Agent和知识库Agent同时执行（并行），而不是串行。
```

**策略三：缓存**

```
相同的查询结果缓存起来，避免重复调用。
```

## 四、产品经理需要掌握什么？

### 4.1 多Agent设计的产品思维

设计多Agent产品时，产品经理需要回答：

**问题一：拆成几个Agent？**

- 太少：每个Agent太重，调试困难
- 太多：协作复杂，Token消耗大
- 推荐：3-7个Agent

**问题二：用哪种组织模式？**

- 任务明确可分解 → Master-Worker
- 任务之间有复杂依赖 → Peer-to-Peer
- 大量独立任务 → Swarm

**问题三：Agent之间怎么通信？**

- 简单传递 → 消息传递
- 复杂协作 → 共享状态
- 混合 → 简单用消息，复杂用共享状态

**问题四：冲突怎么解决？**

- 有明确优先级 → 优先级策略
- 多个独立判断 → 投票策略
- 无法判断 → 人工仲裁

### 4.2 多Agent产品的PRD写法

**模块一：Agent清单**

```
| Agent名称 | 职责 | 模型 | 自主程度 |
|----------|------|------|---------|
| 接待Agent | 接收和初步理解用户消息 | 7B | 低 |
| 查询Agent | 查询工单和订单信息 | 7B | 低 |
| 决策Agent | 决定处理方案 | 70B | 中 |
| 回复Agent | 生成用户回复 | 70B | 中 |
| 质检Agent | 检查回复质量 | 7B | 低 |
```

**模块二：Agent协作流程**

```
用户消息 → 接待Agent → 查询Agent → 决策Agent → 回复Agent → 质检Agent → 发送
```

**模块三：通信协议**

```
- 接待Agent ↔ 查询Agent：消息传递
- 查询Agent → 决策Agent：共享状态
- 决策Agent → 回复Agent：消息传递
- 回复Agent → 质检Agent：共享状态
```

**模块四：冲突解决**

```
- 决策Agent和质检Agent意见不一致 → 质检Agent有一票否决权
- 质检Agent之间意见不一致 → 人工仲裁
```

### 4.3 跟工程师沟通的常用话术

**讨论Agent拆分时：**

> "这个系统拆成几个Agent比较合适？查询和决策能不能合并成一个Agent？"

**讨论通信方式时：**

> "Agent之间用共享状态还是消息传递？共享状态的话，State怎么定义？"

**讨论成本时：**

> "5个Agent串行执行，每轮的Token消耗大概多少？能不能把几个Agent并行执行？"

## 七、真实案例：客户工单系统的多Agent升级

### 7.1 从单Agent到多Agent

**单Agent方案（第七章）：**

```
一个Agent做所有事情：
1. 理解用户消息
2. 查询工单信息
3. 搜索知识库
4. 决定回复内容
5. 生成回复
6. 检查质量
```

问题：Token消耗大、调试困难、单点故障。

**多Agent方案：**

```
5个Agent各司其职：

接待Agent（7B模型）：
  - 接收用户消息
  - 识别意图（查询/投诉/建议）
  - 识别情感（正面/中性/负面）
  - 输出：结构化请求

查询Agent（7B模型）：
  - 查询工单信息
  - 搜索知识库
  - 输出：查询结果

决策Agent（70B模型）：
  - 分析查询结果
  - 决定处理方案
  - 输出：处理方案

回复Agent（70B模型）：
  - 根据处理方案生成回复
  - 控制语气和格式
  - 输出：回复草稿

质检Agent（7B模型）：
  - 检查回复质量
  - 评分（0-100）
  - 输出：通过/不通过 + 评分
```

### 7.2 多Agent架构的优势

**Token消耗降低：**

- 单Agent：每次请求需要携带完整的对话历史 + 所有工具描述 + 所有中间结果 → 5000+ Token
- 多Agent：每个Agent只需要自己的上下文 → 平均每轮1500 Token
- 节省：约70%

**调试更简单：**

- 单Agent：出问题了不知道是哪一步的问题
- 多Agent：每个Agent独立监控，出问题直接定位到具体Agent

**模型选择更灵活：**

- 简单任务（意图分类、质检）用7B模型（便宜、快）
- 复杂任务（决策、回复生成）用70B模型（贵、慢但质量好）
- 成本优化：简单任务不用大模型

### 7.3 效果

上线三个月后：

- 平均响应时间从8秒降到3秒（并行执行 + 小模型处理简单任务）
- 月度Token成本降低40%（小模型替代大模型处理简单任务）
- 回复质量提升（质检Agent独立负责质量把关）
- 调试时间减少60%（定位问题直接到具体Agent）

## 八、常见误区

### 误区一："Agent越多越好"

**真相**：Agent越多，协作越复杂，调试越困难。推荐3-7个Agent。

**实践方法**：先从一个Agent开始，复杂度上来后再拆分。

### 误区二："所有Agent都用同一个模型"

**真相**：简单任务用大模型是浪费，复杂任务用小模型是不够的。

**实践方法**：根据任务复杂度选择模型——简单任务用小模型，复杂任务用大模型。

### 误区三："多Agent一定比单Agent好"

**真相**：多Agent的开发和维护成本更高。如果任务简单，单Agent就够了。

**实践方法**：用复杂度评估——任务简单用单Agent，任务复杂用多Agent。

### 误区四："Agent之间通信越快越好"

**真相**：通信快不代表协作好。Agent之间的通信频率过高会增加延迟和成本。

**实践方法**：减少不必要的通信，只在关键节点交换信息。

### 误区五："多Agent系统更难测试"

**真相**：确实更难测试，但每个Agent可以单独测试（单元测试），这反而是优势。

**实践方法**：先测试每个Agent的单元测试，再测试整体的集成测试。

### 误区六："多Agent就是并行执行"

**真相**：多Agent可以是串行、并行或混合。不是所有Agent都能并行。

**实践方法**：分析Agent之间的依赖关系，只有没有依赖的Agent才能并行。

### 误区七："多Agent不需要协调"

**真相**：多Agent更需要协调。没有协调的Agent团队就是一盘散沙。

**实践方法**：设计明确的协调机制——Master-Worker用Planner，Peer-to-Peer用消息协议。

## 九、面试题

### 初级

**Q1：什么是多Agent系统？它和单Agent有什么区别？**

**参考答案：**

多Agent系统是由多个Agent组成的系统，每个Agent有自己的职责和技能，它们协作完成复杂的任务。

和单Agent的区别：

- 单Agent：一个Agent做所有事，Token消耗大，调试困难
- 多Agent：多个Agent各司其职，每个Agent可以用小模型，调试简单

类比：单Agent像一个全能选手，多Agent像一个专业团队。

---

**Q2：多Agent有哪几种组织模式？**

**参考答案：**

1. Master-Worker：一个Master负责任务分解和协调，多个Worker负责执行。适合任务可以明确分解的场景。
2. Peer-to-Peer：所有Agent地位平等，可以自由通信和协作。适合任务之间有复杂依赖的场景。
3. Swarm：大量Agent独立处理相似任务，互不干扰。适合大量独立任务的场景。

---

### 中级

**Q3：多Agent系统中，Agent之间怎么通信？各有什么优缺点？**

**参考答案：**

三种通信方式：

1. **共享状态**：所有Agent读写同一个状态对象。优点是简单直接、信息传递高效。缺点是并发访问需要锁机制。
2. **消息传递**：Agent之间通过消息通信。优点是解耦、易于调试。缺点是通信延迟高。
3. **混合模式**：简单任务用消息传递，复杂协作用共享状态。

---

**Q4：如何评估多Agent系统的效果？**

**参考答案：**

从四个维度评估：

1. **效果**：整体任务完成的质量（回复准确率、用户满意度）
2. **效率**：整体执行时间（从用户提问到收到回复的延迟）
3. **成本**：Token消耗和模型费用
4. **可维护性**：调试难度、Agent数量、协作复杂度

---

### 高级

**Q5：你设计了一个多Agent客服系统，发现质检Agent经常否决回复Agent的回复，导致用户等待时间过长。你怎么办？**

**参考答案：**

**问题分析：**

质检Agent频繁否决说明回复质量不稳定。可能的原因：

1. 质检标准太严格
2. 回复Agent的能力不够
3. 回复Agent和质检Agent的标准不一致

**解决方案：**

1. **调整质检标准**：降低质检阈值，从85分降到75分
2. **给回复Agent反馈**：质检Agent否决时，告诉回复Agent哪里不好，让它改进
3. **增加一轮迭代**：回复Agent根据质检反馈重新生成回复（最多2轮）
4. **分级质检**：简单回复自动通过，复杂回复才需要质检

**长期方案：**

- 用质检Agent的否决数据训练回复Agent
- 建立"回复-质检"配对数据，让回复Agent学习质检Agent的标准

---

**Q6：设计一个多Agent系统来处理复杂的金融分析任务，需要考虑哪些因素？**

**参考答案：**

**Agent设计：**

1. 数据采集Agent：从多个数据源获取数据
2. 数据清洗Agent：清洗和标准化数据
3. 分析Agent：执行财务分析（比率分析、趋势分析、对比分析）
4. 报告Agent：生成分析报告
5. 合规Agent：检查报告是否符合合规要求
6. 人工审核Agent：最终审核

**关键因素：**

1. **数据安全**：金融数据敏感，需要加密和权限控制
2. **可追溯性**：每一步分析都需要有审计日志
3. **准确性**：金融分析对准确性要求极高，需要多重验证
4. **合规性**：报告需要符合监管要求
5. **实时性**：市场数据需要实时更新
6. **模型选择**：分析用大模型，合规检查用小模型
7. **冲突解决**：分析Agent和合规Agent意见不一致时，合规Agent有一票否决权

## 十、本章总结

本章我们系统学习了多Agent系统的核心概念和应用：

1. **为什么需要多Agent**：单Agent有上下文窗口、注意力、容错、调试的限制
2. **三种组织模式**：Master-Worker（主从）、Peer-to-Peer（对等）、Swarm（蜂群）
3. **Agent角色设计**：职责、技能、输入输出
4. **通信协议**：共享状态、消息传递、混合模式
5. **冲突解决**：优先级、投票、人工仲裁
6. **成本优化**：小模型做简单任务、并行执行、缓存
7. **客户工单系统多Agent升级**：从单Agent到5个Agent，Token成本降低40%，响应时间从8秒降到3秒
8. **7个常见误区**：Agent不是越多越好、不是所有Agent都用同一个模型、不是多Agent一定更好

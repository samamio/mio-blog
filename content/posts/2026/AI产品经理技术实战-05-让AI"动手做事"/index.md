## 一、背景：从"聊天"到"做事"的跨越

### 1.1 大模型的"先天不足"

大模型有一个根本性的限制：**它只存在于自己的参数和输入的上下文中。**

换句话说：

- 模型知道"怎么查物流"——它读过无数关于物流查询的文章
- 模型不知道"订单12345的物流是什么"——因为它没有访问订单数据库的权限
- 模型可以"写"一段查询物流的代码——但它不能"执行"这段代码

这就像一位百科全书式的学者：他什么都知道，但他没有电话、没有电脑、没有出门的时间。他只能"告诉你"怎么做，不能"替你"去做。

**Function Calling就是给这位学者装上电话和电脑。**

### 1.2 Function Calling的诞生

2023年，OpenAI在GPT-3.5 Turbo中首次引入了Function Calling功能。在此之前，如果你想让模型帮你做点什么（查天气、查订单、发邮件），模型只能"告诉你怎么做"。

引入Function Calling之后，模型可以做三件事：

1. **判断**：用户的问题是否需要调用工具
2. **选择**：如果需要调用工具，选择哪个工具
3. **传参**：从用户的话语中提取工具需要的参数

### 1.3 为什么Function Calling对产品经理至关重要？

因为**Function Calling是AI从"聊天机器人"变成"智能助手"的关键一跃。**

没有Function Calling的AI：
- 能回答"怎么查物流"
- 不能回答"订单12345的物流到哪了"

有Function Calling的AI：
- 能回答"订单12345的物流已到北京中转站，预计明天送达"

前者是"顾问"，后者是"助手"。用户需要的不是顾问，是助手。

## 二、核心原理：Function Calling是怎么工作的？

### 2.1 Function Calling的工作流程

让我们用一个具体的例子来理解。

**场景：** 用户问"帮我查一下订单12345的物流"

**完整流程：**

```
第1步：用户发送消息
"帮我查一下订单12345的物流"
       │
       ▼
第2步：应用层发送请求给模型
系统提示词：你是一个客服助手，可以调用以下工具：
  - get_order_info(order_id): 查询订单信息
  - get_tracking(tracking_id): 查询物流信息
  - transfer_to_human(reason): 转接人工客服
       │
       ▼
第3步：模型分析意图
模型思考："用户想查物流，需要调用get_tracking工具，
          但tracking_id是订单号12345，
          我需要先调用get_order_info获取tracking_id"
       │
       ▼
第4步：模型返回Function Call请求
{
  "function": "get_order_info",
  "arguments": {
    "order_id": "12345"
  }
}
       │
       ▼
第5步：应用层执行函数
调用get_order_info("12345")
返回：{
  "order_id": "12345",
  "tracking_id": "SF1234567890",
  "status": "shipped"
}
       │
       ▼
第6步：应用层将结果返回给模型
系统消息：{"function_name": "get_order_info", "result": {"order_id": "12345", "tracking_id": "SF1234567890", "status": "shipped"}}
       │
       ▼
第7步：模型生成最终回复
"您的订单12345已发货，物流单号SF1234567890。
  当前状态：运输中。预计明天送达。"
```

### 2.2 模型是如何决定调用哪个工具的？

模型做这个决定的方式很简单：**匹配**。

当应用层发送请求时，会同时发送所有可用工具的"描述"。模型会根据用户的消息和工具描述，选择最匹配的那个。

**工具描述示例：**

```json
{
  "name": "get_tracking",
  "description": "查询物流信息。根据物流单号获取包裹的当前位置和预计送达时间。",
  "parameters": {
    "tracking_id": {
      "type": "string",
      "description": "物流单号，例如SF1234567890",
      "required": true
    }
  }
}
```

模型看到工具描述后，会做这样的匹配：

```
用户消息："帮我查一下订单12345的物流"
工具描述："查询物流信息。根据物流单号获取包裹当前位置..."
匹配结果：用户想查物流 → 调用get_tracking工具
参数提取：物流单号 = SF1234567890（如果用户直接提供了）
```

### 2.3 参数提取：从自然语言到结构化数据

这是Function Calling最核心的能力之一：**从用户的自然语言中提取结构化的参数。**

举例：

```
用户说："帮我查一下订单12345的物流"
模型提取：{"order_id": "12345"}
```

```
用户说："我昨天买的蓝色围巾到哪了，订单号是ORD-2024-001"
模型提取：{"order_id": "ORD-2024-001"}
```

```
用户说："查物流"
模型判断：缺少order_id参数，需要追问用户
```

**关键点：模型不是"理解"了订单号，而是通过模式匹配"认出"了订单号的格式。**

这解释了为什么工具描述中的参数描述非常重要——它告诉模型"这个参数长什么样"。

### 2.4 Function Calling vs RAG：两者的区别和配合

很多产品经理会混淆Function Calling和RAG。它们是不同的技术，但经常配合使用。

**Function Calling：** 让模型"做动作"

- 查数据库
- 发邮件
- 创建订单
- 调用API

**RAG：** 让模型"获取知识"

- 查询知识库文档
- 获取最新新闻
- 读取产品手册

**配合使用示例：**

```
用户："我的订单12345有问题，我要退款"

Step 1（Function Calling）：查询订单信息
→ 返回订单详情：金额500元，已发货3天

Step 2（RAG）：查询退款政策
→ 返回政策：已发货订单可在签收后7天内申请退款

Step 3（Function Calling）：创建退款申请
→ 返回退款单号：RF-2024-001

Step 4（模型生成回复）：
"已为您创建退款申请，退款单号RF-2024-001。
  由于订单已发货，需要在签收后7天内申请。
  退款将在3-5个工作日内原路退回。"
```

## 三、企业应用：工具描述的最佳实践

### 3.1 工具描述的写法

工具描述是Function Calling的灵魂。**描述写得好，模型才会调、会调对。**

**差的描述：**

```json
{
  "name": "get_order_info",
  "description": "获取订单信息",
  "parameters": {
    "order_id": {
      "type": "string",
      "description": "订单号"
    }
  }
}
```

问题：
- "获取订单信息"太模糊——什么算"信息"？
- "订单号"没有说明格式——是数字？字母？还是两者都有？

**好的描述：**

```json
{
  "name": "get_order_info",
  "description": "根据订单号查询订单的详细信息，包括订单状态、商品列表、收货地址、物流单号等。如果订单号不存在或无效，返回错误信息。",
  "parameters": {
    "order_id": {
      "type": "string",
      "description": "订单号，格式为ORD-YYYY-NNN，例如ORD-2024-001。可以在用户的订单确认邮件或订单页面找到。",
      "required": true
    }
  }
}
```

改进点：
- 说明了返回什么信息（订单状态、商品列表等）
- 说明了出错时怎么处理
- 说明了订单号的格式和在哪里能找到

### 3.2 工具命名的规范

工具命名看似小事，但对模型的理解有很大影响。

**好的命名原则：**

| 原则 | 示例 | 说明 |
|------|------|------|
| 动词+名词 | `get_order_info` | 清楚表达工具的用途 |
| 使用英文 | `search_products` | 大多数模型对英文工具名的理解更好 |
| 小写+下划线 | `transfer_to_human` | 保持一致的命名风格 |
| 避免缩写 | `get_customer_email` 而非 `get_cus_eml` | 让模型更容易理解 |
| 避免歧义 | `create_ticket` 而非 `create` | 明确是哪个create |

### 3.3 工具数量的管理

当工具数量较少（<10个）时，直接把所有工具描述发给模型就行。

当工具数量较多（>20个）时，需要优化：

**策略一：工具分组**

```
用户问物流相关问题 → 只发送物流相关的工具描述
用户问订单相关问题 → 只发送订单相关的工具描述
```

**策略二：工具检索**

先用一个小模型（或关键词匹配）根据用户的问题，检索最相关的3-5个工具，只把这3-5个工具的描述发给大模型。

**策略三：工具分层**

- 第一层：大类工具（5-10个），负责"路由"——判断用户需要什么类型的帮助
- 第二层：具体工具（每个大类下有多个具体工具），由大类工具调用

### 3.4 客户工单系统的工具设计

让我们回到贯穿全书的案例。

**工具清单：**

```
1. query_ticket(ticket_id)
   查询指定工单的详细信息（状态、分类、处理进度等）

2. create_ticket(user_id, category, description, priority)
   创建一个新的工单

3. update_ticket(ticket_id, fields)
   更新工单的指定字段（状态、分类、备注等）

4. search_tickets(user_id, keywords)
   搜索用户的历史工单

5. transfer_to_human(ticket_id, reason)
   将工单转接给人工客服

6. get_suggested_reply(ticket_id)
   根据工单内容，从知识库获取建议回复

7. close_ticket(ticket_id, resolution_note)
   关闭工单并记录解决方案

8. get_user_profile(user_id)
   查询用户的基本信息（等级、历史工单数、满意度等）
```

**工具描述示例（query_ticket）：**

```json
{
  "name": "query_ticket",
  "description": "查询指定工单的详细信息。返回内容包括：工单状态（待处理/处理中/已解决/已关闭）、工单分类、创建时间、用户描述、处理记录、负责的客服人员。如果工单不存在，返回'工单不存在'。",
  "parameters": {
    "ticket_id": {
      "type": "string",
      "description": "工单ID，格式为TK-XXXXX，例如TK-12345。可以从用户消息或历史工单记录中获取。",
      "required": true
    }
  }
}
```

## 四、产品经理需要掌握什么？

### 4.1 工具设计的产品思维

作为产品经理，你不需要写工具的代码，但你需要设计工具的"接口"——也就是工具描述。

**设计工具的三个问题：**

1. **这个工具用户会怎么用？**
   - 用户说"查订单"时，他会提供订单号吗？还是只提供手机号？
   - 如果参数不全，模型应该怎么处理？

2. **这个工具返回的数据，模型能理解吗？**
   - 如果返回的是JSON，模型能正确解析吗？
   - 如果返回的是HTML，模型可能会把HTML标签也输出给用户——需要预处理

3. **这个工具的安全边界在哪里？**
   - 哪些操作不能让模型自动执行？（如删除数据、修改价格）
   - 哪些操作需要人工确认？（如退款超过1000元）

### 4.2 工具设计的PRD写法

AI产品的PRD中，Function Calling部分需要描述：

**模块一：工具清单**

```
工具名称 | 用途 | 输入参数 | 返回内容 | 调用时机
---------|------|---------|---------|--------
query_ticket | 查询工单详情 | ticket_id | 工单信息JSON | 用户询问工单状态时
create_ticket | 创建新工单 | user_id, category, description | 工单ID | 用户提出新问题时
transfer_to_human | 转接人工 | ticket_id, reason | 转接结果 | 模型判断无法自动处理时
```

**模块二：调用流程**

```
用户消息 → 意图判断 → 是否需要调用工具？
  → 是 → 选择工具 → 提取参数 → 调用工具 → 获取结果 → 生成回复
  → 否 → 直接生成回复
```

**模块三：错误处理**

```
工具调用失败的情况：
1. 参数缺失 → 模型追问用户
2. 参数格式错误 → 模型告知用户格式要求
3. 工具返回错误 → 模型用友好语言转述错误
4. 工具超时 → 告知用户"正在处理中，请稍后再试"
```

### 4.3 跟工程师沟通的常用话术

**讨论工具设计时：**

> "这个工具需要哪些参数？如果用户没说订单号，模型能不能从上下文里找到？"

**讨论错误处理时：**

> "如果工具调用失败了，用户看到的是什么？是技术错误信息，还是友好的提示？"

**讨论安全边界时：**

> "哪些操作不能让模型自动执行？比如退款超过1000元，是不是需要人工确认？"

## 五、研发如何实现：Function Calling的工程实现

### 5.1 完整的请求-响应流程

**应用层代码逻辑（伪代码）：**

```python
def handle_user_message(user_message, user_id):
    # Step 1: 获取可用的工具列表
    tools = get_available_tools(user_id)
    
    # Step 2: 发送请求给模型
    response = llm.chat(
        messages=[
            {"role": "system", "content": SYSTEM_PROMPT},
            {"role": "user", "content": user_message}
        ],
        tools=tools,
        temperature=0.1  # 工具调用需要低随机性
    )
    
    # Step 3: 检查模型是否要调用工具
    if response.has_tool_calls():
        for tool_call in response.tool_calls:
            # 执行工具
            result = execute_tool(tool_call.function_name, tool_call.arguments)
            
            # 将结果返回给模型
            response = llm.chat(
                messages=response.messages + [
                    {"role": "tool", "tool_call_id": tool_call.id, "content": result}
                ],
                tools=tools
            )
        
        # 返回模型的最终回复
        return response.final_message
    else:
        # 模型直接回复，不调用工具
        return response.message
```

### 5.2 多轮工具调用的处理

有些任务需要调用多个工具。比如：

```
用户："帮我查一下订单12345的物流"

模型第一次调用：get_order_info(order_id="12345")
返回：{"tracking_id": "SF1234567890", "status": "shipped"}

模型第二次调用：get_tracking(tracking_id="SF1234567890")
返回：{"location": "北京中转站", "eta": "2024-06-15"}

模型最终回复："您的订单12345已到达北京中转站，预计6月15日送达。"
```

**关键设计点：**

1. **最大调用次数限制**：防止模型陷入无限循环（一直调工具一直调工具）
2. **调用超时**：每个工具调用不能超过5秒
3. **调用链日志**：记录每一次调用的输入、输出、耗时，方便调试

### 5.3 工具调用的安全性

**沙箱机制：**

```
不允许模型自动执行的工具：
- delete_ticket()  # 删除工单
- refund_money()  # 退款（超过一定金额需要人工确认）
- change_price()  # 修改商品价格
- export_data()   # 导出数据（可能包含隐私信息）
```

**审批机制：**

```
需要人工确认的操作：
1. 模型生成操作请求
2. 系统判断操作敏感度
3. 敏感操作 → 弹出人工确认界面
4. 不敏感操作 → 自动执行
```

## 七、真实案例：某外卖平台的"AI订餐助手"

### 7.1 背景

某外卖平台有5000万用户，每天收到大量客服咨询。其中约40%是订单相关问题：

- "我的订单到哪了？"
- "帮我退一下单"
- "配送费怎么收了这么多？"
- "我想改一下收货地址"

原来的方案：关键词匹配 + FAQ。用户体验很差——用户说"我的东西还在路上但我不想吃了"，机器人识别不了，直接转人工。

### 7.2 设计方案

设计了一组Function Calling工具：

```
1. get_order_status(order_id)  # 查询订单状态
2. cancel_order(order_id, reason)  # 取消订单
3. modify_address(order_id, new_address)  # 修改地址
4. refund_request(order_id, amount, reason)  # 申请退款
5. contact_rider(order_id)  # 联系骑手
6. contact_support(order_id, issue)  # 转人工客服
```

**工具描述示例（cancel_order）：**

```json
{
  "name": "cancel_order",
  "description": "取消指定的订单。只能在订单状态为'待接单'时取消。如果商家已接单，则无法取消，需要转人工客服处理。取消成功后，款项将在1-3个工作日内退回原支付账户。",
  "parameters": {
    "order_id": {
      "type": "string",
      "description": "订单号，格式为数字，例如1234567890",
      "required": true
    },
    "reason": {
      "type": "string",
      "description": "取消原因（可选），例如'不想吃了'、'下单错误'",
      "required": false
    }
  }
}
```

### 7.3 安全设计

```
自动执行的工具（低风险）：
- get_order_status  # 只读操作
- contact_rider     # 联系骑手

需要人工确认的工具（中风险）：
- cancel_order      # 取消订单涉及退款
- modify_address    # 修改地址影响配送

禁止自动执行的工具（高风险）：
- refund_request    # 退款超过一定金额需要人工审批
- contact_support   # 转人工由系统自动路由
```

### 7.4 效果

上线三个月后：

- 订单相关咨询的自动解决率从25%提升到65%
- 人工客服的订单相关工作量减少了50%
- 用户对AI助手的满意度从55%提升到78%

### 7.5 关键经验

1. **工具描述要包含"边界条件"**：比如`cancel_order`的描述中明确说了"只能在待接单时取消"，这让模型在商家已接单时不会强行调用，而是转人工。
2. **安全分级很重要**：不是所有工具都能让模型自动执行，尤其是涉及金钱的操作。
3. **参数提取需要示例**：在工具描述中给出参数格式的示例（如"订单号格式为数字，例如1234567890"），能显著提高参数提取的准确率。

## 八、常见误区

### 误区一："工具越多越好"

**真相**：工具太多会增加模型的"认知负担"。模型需要同时理解几十个工具的用途和参数，容易选错。

**实践方法**：每个产品场景下，工具数量控制在10-15个以内。超过这个数量，用工具分组或检索策略。

### 误区二："工具描述写越长越好"

**真相**：工具描述需要详细，但不是越长越好。过长的描述会消耗更多Token，增加成本。

**实践方法**：控制在2-3句话内说清楚工具的用途、输入输出、边界条件。

### 误区三："模型不会搞错工具"

**真相**：模型确实会搞错工具，特别是当两个工具的描述很相似时。

举例：`search_products(keyword)` 和 `search_orders(keyword)` 的描述如果不够清晰，模型可能把"查订单"匹配到"搜商品"。

**实践方法**：工具描述要避免歧义，必要时在描述中加入"不要调用此工具的情况"。

### 误区四："工具调用失败了重试就行"

**真相**：有些错误重试没有意义。比如参数错误，重试只会得到同样的错误。

**实践方法**：区分"可重试错误"（超时、系统错误）和"不可重试错误"（参数错误、权限不足）。

### 误区五："Function Calling能替代所有API"

**真相**：Function Calling适合"模型理解意图后调用"的场景。但对于需要精确控制的场景（如定时任务、批量处理），传统API调用更合适。

**实践方法**：判断标准——如果操作的触发条件需要"理解自然语言"，用Function Calling。否则用传统API。

### 误区六："模型提取的参数一定是正确的"

**真相**：模型可能会提取错参数。比如用户说"订单12345和67890"，模型可能只提取了一个。

**实践方法**：在应用层做参数校验——检查参数格式、范围、合理性。

### 误区七："Function Calling是万能的"

**真相**：Function Calling只能调用"有明确输入输出"的工具。对于那些需要复杂交互或多步骤确认的操作，Function Calling不够用。

**实践方法**：对于复杂操作，设计多轮对话流程——Function Calling只负责最简单的"一问一答"式操作。

## 九、面试题

### 初级

**Q1：什么是Function Calling？它解决了什么问题？**

**参考答案：**

Function Calling是大模型的一项能力，让模型能够：
1. 判断用户的问题是否需要调用外部工具
2. 选择正确的工具
3. 从用户的自然语言中提取工具需要的参数

它解决的问题是：大模型只能"说"不能"做"。有了Function Calling，模型可以调用数据库、API、工具来完成实际操作，从"聊天机器人"变成"智能助手"。

---

**Q2：如果一个用户说"查一下我的订单"，模型怎么知道要调用哪个工具？**

**参考答案：**

模型通过以下方式判断：

1. **工具描述匹配**：应用层会把所有可用工具的描述发给模型。模型根据"查订单"这个意图，匹配到`get_order_info`工具。
2. **参数提取**：模型尝试从用户消息中提取参数（订单号）。如果用户没有提供订单号，模型会追问："请问您的订单号是多少？"
3. **上下文理解**：如果用户之前提到过订单号，模型会从对话历史中提取。

---

### 中级

**Q3：你设计了10个工具，但发现模型经常调用错误的工具。可能的原因是什么？怎么解决？**

**参考答案：**

可能的原因：

1. **工具描述太相似**：两个工具的描述差别太小，模型分不清。比如`search_products`和`search_orders`的描述都只有"搜索..."。
   - 解决：重写工具描述，加入明确的区分信息。

2. **工具太多**：10个工具同时发给模型，增加了选择的复杂度。
   - 解决：用工具分组策略——根据用户问题的类型，只发送相关的3-5个工具描述。

3. **参数描述不清晰**：模型不知道哪个参数对应哪个工具。
   - 解决：在参数描述中加入格式示例。

4. **缺少边界条件**：模型不知道"什么时候不该调用这个工具"。
   - 解决：在工具描述中加入"不要调用此工具的情况"。

---

**Q4：Function Calling和RAG有什么区别？它们怎么配合？**

**参考答案：**

**区别：**

- Function Calling让模型"做动作"——查数据库、发邮件、创建订单
- RAG让模型"获取知识"——从你的文档库中检索相关信息

**配合方式：**

一个典型的AI客服流程：

1. Function Calling：查询用户的订单信息（从数据库）
2. RAG：查询退款政策文档（从知识库）
3. Function Calling：根据政策和订单信息，创建退款申请
4. 模型生成最终回复，综合订单信息和退款政策

Function Calling负责"操作数据"，RAG负责"理解知识"。两者配合才能实现完整的智能客服。

---

### 高级

**Q5：你负责一个AI客服产品，发现模型在调用工具时有以下问题：①经常遗漏参数 ②有时调用不相关的工具 ③工具返回结果后模型不理解。你如何解决？**

**参考答案：**

针对三个问题分别解决：

**问题①：经常遗漏参数**
- 在工具描述中明确标注哪些参数是必需的
- 在参数描述中加入格式示例（如"订单号格式为ORD-YYYY-NNN"）
- 在系统提示词中强调"如果参数缺失，先追问用户再调用工具"
- 在应用层做参数校验，缺失参数时不调用工具

**问题②：调用不相关的工具**
- 优化工具描述，确保每个工具的描述有明确的区分特征
- 减少同时发送的工具数量，用工具分组策略
- 在工具描述中加入"不要调用此工具的情况"

**问题③：工具返回结果后模型不理解**
- 优化工具返回数据的格式，用自然语言描述返回结果
- 在系统提示词中说明每个工具的返回格式
- 对于复杂返回结果，先做预处理再发给模型

**系统性方案**：
建立工具调用的完整测试集，覆盖以上三种场景，每次优化后用测试集验证。

**Q6：设计一个AI订餐助手，需要哪些工具？请列出工具清单、参数设计、安全策略。**

**参考答案：**

**工具清单：**

| 工具名 | 用途 | 必需参数 | 可选参数 |
|--------|------|---------|---------|
| browse_menu | 浏览菜单 | 无 | category（分类） |
| search_dish | 搜索菜品 | keyword | restaurant_id |
| get_restaurant_info | 获取餐厅信息 | restaurant_id | - |
| add_to_cart | 加入购物车 | dish_id, quantity | restaurant_id |
| remove_from_cart | 移出购物车 | cart_item_id | - |
| place_order | 下单 | address_id, payment_method | coupon_code |
| get_order_status | 查询订单状态 | order_id | - |
| cancel_order | 取消订单 | order_id | reason |
| modify_order | 修改订单 | order_id, modifications | - |
| contact_support | 联系客服 | issue_description | order_id |

**安全策略：**

- 只读操作（browse_menu、search_dish、get_order_status）：自动执行
- 写入操作（add_to_cart、place_order）：需要用户确认
- 敏感操作（cancel_order、modify_order）：需要用户确认 + 订单状态检查
- 禁止自动执行（contact_support）：由系统自动路由

## 十、本章总结

本章我们学习了Function Calling的核心原理和最佳实践：

1. **Function Calling的本质**：给大模型装上"手脚"，让它能从"说"变成"做"
2. **工作流程**：用户消息 → 模型意图分析 → 选择工具 → 提取参数 → 执行工具 → 获取结果 → 生成回复
3. **工具描述是关键**：描述写得好，模型才会调、会调对。包含用途、参数格式、边界条件
4. **工具数量管理**：10个以内直接发送，超过20个用分组或检索策略
5. **安全策略**：按风险分级，只读操作自动执行，涉及金钱的操作需要人工确认
6. **客户工单系统工具设计**：8个工具覆盖查询、创建、更新、搜索、转接、回复建议、关闭、用户信息
7. **外卖平台案例**：从25%到65%的自动解决率提升

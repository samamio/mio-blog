# 第6章 RAG——让AI拥有你的知识库

## 本章目标

第五章我们让AI学会了"动手"——通过Function Calling调用外部工具。但还有一个根本问题：**AI的知识是有限的。**

即使是最强的大模型，它的知识也有截止时间。而且，即使没有截止时间问题，模型也不可能知道你公司的内部资料——产品手册、FAQ文档、操作指南、历史工单解决方案。

RAG（Retrieval-Augmented Generation，检索增强生成）解决的就是这个问题：**让AI在回答之前，先"查一下资料"。**

学习完本章后，你将能够：

1. **理解RAG的核心原理**——为什么"查一下再用"比"记住全部"更好
2. **掌握文档切分策略**——Chunk的大小、重叠、层级设计对召回质量的影响
3. **理解Embedding和向量搜索**——相似度度量是什么、有什么陷阱
4. **设计检索策略**——同义搜索、关键词混合搜索、重排序（Rerank）
5. **评估RAG系统**——召回率、精确率、答案相关性三大指标
6. **完成客户工单系统的RAG搭建**——处理内部操作手册

## 一、背景：为什么需要RAG？

### 1.1 大模型的"知识天花板"

大模型有一个根本性的限制：**它的知识仅限于训练数据。**

这意味着：

1. **知识有截止时间**：模型训练数据截止到什么时候，它就只知道到那个时候的事。2024年之后的新闻，它不知道。
2. **不知道你的内部数据**：你公司的产品手册、操作指南、客户数据——模型从来没见过。
3. **知识可能过时**：产品更新了，旧的功能下线了，新的功能上线了。模型的训练数据不会自动更新。
4. **无法溯源**：模型说"根据XX文档"，但它其实没有看过那份文档。它只是在"编造"一种看起来很可信的说法。

**这就是RAG诞生的原因。**

### 1.2 RAG的核心洞察

RAG的核心洞察非常简单，但非常强大：

> **与其让模型"记住"所有知识，不如让模型在需要的时候"查一下"。**

类比：

- **没有RAG的模型**：像一个背了很多书的学生。书背得越多越好，但总有背不到的。而且他不能带任何参考资料进考场。
- **有RAG的模型**：像一个可以带参考资料进考场的学生。他不需要记住所有知识，只需要知道"去哪里查"和"怎么理解查到的内容"。

第二种方式显然更灵活、更可靠。

### 1.3 RAG的诞生

2020年，Meta（原Facebook）的论文《Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks》首次提出了RAG的概念。

论文的核心发现：**在生成任务中，让模型从外部知识库检索相关信息，再基于检索到的信息生成回答，效果显著优于纯生成模型。**

从此，RAG成为了AI产品中最常用的技术架构之一。

### 1.4 RAG vs 微调：什么时候用哪个？

很多产品经理会混淆RAG和微调。它们都能让模型"知道"你的数据，但方式完全不同。

| 维度 | RAG | 微调 |
|------|-----|------|
| 方式 | 检索时临时"借"知识 | 训练时永久"学"知识 |
| 更新速度 | 即时（改文档就生效） | 慢（重新训练需要数小时） |
| 成本 | 低（只需要向量数据库） | 高（需要训练数据和算力） |
| 知识量 | 可以处理海量文档 | 只能处理有限的训练数据 |
| 可溯源性 | 好（可以追溯到原始文档） | 差（不知道知识从哪来） |
| 适用场景 | 知识频繁更新、知识量大 | 学习特定风格/格式/专业知识 |

**简单规则：**

- 知识频繁更新 → RAG
- 知识量大（>1000页） → RAG
- 需要溯源 → RAG
- 学习特定格式/风格 → 微调
- 领域专业知识（模型完全不懂的） → 微调
- 两者结合 → RAG + 微调（最佳实践）

## 二、核心原理：RAG是怎么工作的？

### 2.1 RAG的完整流程

```
用户提问："怎么重置密码？"
       │
       ▼
┌─────────────────────────────┐
│  Step 1: 查询改写            │
│  原始问题 → 优化后的问题      │
│  "怎么重置密码？"            │
│  → "用户账户密码重置操作步骤"  │
└──────────┬──────────────────┘
           │
           ▼
┌─────────────────────────────┐
│  Step 2: 向量化              │
│  将问题转换为Embedding向量    │
│  "怎么重置密码" → [0.23, -0.45, ...] │
└──────────┬──────────────────┘
           │
           ▼
┌─────────────────────────────┐
│  Step 3: 向量搜索            │
│  在向量数据库中搜索最相似的文档片段 │
│  返回Top-K个最相关的chunk    │
└──────────┬──────────────────┘
           │
           ▼
┌─────────────────────────────┐
│  Step 4: 重排序（可选）       │
│  对检索结果做更精细的相关度排序 │
│  选出最相关的3个chunk        │
└──────────┬──────────────────┘
           │
           ▼
┌─────────────────────────────┐
│  Step 5: 组装Prompt          │
│  系统提示词 + 检索到的文档 + 用户问题 │
└──────────┬──────────────────┘
           │
           ▼
┌─────────────────────────────┐
│  Step 6: 模型生成回答         │
│  基于检索到的文档，生成准确回答  │
│  "您可以按照以下步骤重置密码：  │
│   1. 登录页面点击'忘记密码'     │
│   2. 输入注册邮箱              │
│   3. 查收邮件并点击重置链接    │
│   4. 设置新密码"              │
└─────────────────────────────┘
```

### 2.2 文档切分（Chunking）：RAG的第一道关口

文档切分是RAG中**最重要但也最容易被忽视**的步骤。

**为什么需要切分？**

因为模型的上下文窗口是有限的。你不能把整本产品手册（500页）一次性塞给模型。你需要把文档切成小块（chunks），每次只检索最相关的那几块。

**切分的核心问题：**

1. **Chunk大小**：每块切多长？
2. **重叠（Overlap）**：相邻两块之间有多少重叠？
3. **切分粒度**：按什么切？按段落？按章节？按句子？

#### Chunk大小的选择

| Chunk大小 | 适用场景 | 优点 | 缺点 |
|-----------|---------|------|------|
| 小（100-300字） | 精确查询（"XX功能的开关在哪"） | 检索精准，Token消耗少 | 可能丢失上下文 |
| 中（300-800字） | 通用场景 | 平衡精度和上下文 | 无明显缺点 |
| 大（800-2000字） | 需要完整理解（"XX产品的整体架构"） | 保留完整上下文 | 检索不精准，Token消耗多 |

**推荐：默认500字左右，重叠100-150字。**

#### 重叠（Overlap）的重要性

重叠是指相邻两个chunk之间有重复的内容。

举例：

```
Chunk 1: "...第1节 安装步骤 第2节 配置步骤..."
Chunk 2: "...第1节 安装步骤 第2节 配置步骤 第3节 使用指南..."
```

Chunk 1和Chunk 2有"第1节 安装步骤 第2节 配置步骤"这部分重叠。

**为什么需要重叠？**

因为如果切分正好把一个完整的段落截断了，没有重叠的话，这个段落的两部分就可能被分到两个chunk里，检索时只能找到一部分，信息不完整。

**推荐重叠：chunk大小的20%-30%。**

#### 切分粒度的选择

| 粒度 | 切分方式 | 适用场景 |
|------|---------|---------|
| 按句子 | 每个句子一个chunk | 精确查询，FAQ |
| 按段落 | 每个段落一个chunk | 通用场景 |
| 按章节 | 每个章节一个chunk | 需要完整理解的场景 |
| 自适应 | 根据文档结构智能切分 | 复杂文档（最佳实践） |

**最佳实践：自适应切分。**

对于结构化的文档（如产品手册、操作指南），按章节切分效果最好。因为每个章节本身就是一个语义完整的单元。

对于非结构化文档（如会议纪要、聊天记录），按段落或句子切分更合适。

### 2.3 Embedding和向量搜索：RAG的"搜索引擎"

#### Embedding回顾

上一章我们简要介绍了Embedding。在这里，我们深入理解它在RAG中的作用。

Embedding把文字转换成向量。向量有一个重要性质：**语义相近的文字，它们的向量在空间中距离近。**

```
"怎么重置密码" 的向量 ≈ [0.23, -0.45, 0.78, ...]
"忘记密码了怎么办" 的向量 ≈ [0.21, -0.43, 0.76, ...]
"如何注册新账号" 的向量 ≈ [0.89, 0.34, -0.12, ...]

前两个向量很接近（语义相近）
前两个向量和第三个向量差别很大（语义不同）
```

#### 相似度度量

怎么判断两个向量"有多接近"？最常用的方法是**余弦相似度（Cosine Similarity）**。

余弦相似度的取值范围是[-1, 1]：

- 1：完全相同（方向完全一致）
- 0：完全不相关（方向垂直）
- -1：完全相反

在RAG中，我们通常取0-1的范围。相似度越高，表示文档和问题越相关。

**产品经理需要理解什么？**

1. 余弦相似度不是"绝对正确"的——它只是一个近似度量
2. 不同的Embedding模型，相似度计算的准确度不同
3. 相似度阈值需要调优——设太高会漏掉相关内容，设太低会混入不相关内容

#### 向量数据库

向量数据库是用来存储和搜索Embedding向量的系统。

**主流向量数据库：**

| 数据库 | 特点 | 适用场景 |
|--------|------|---------|
| Milvus | 开源，可扩展性强 | 大规模生产环境 |
| Pinecone | 托管服务，开箱即用 | 快速原型、中小规模 |
| Weaviate | 开源+云端，支持混合搜索 | 需要混合搜索的场景 |
| FAISS | Facebook出品，速度快 | 研究和原型开发 |
| Chroma | 轻量级，适合嵌入式 | 小型项目、本地开发 |

**产品经理选型建议：**

- 初创团队/快速验证 → Pinecone（托管服务，不用管运维）
- 有技术团队/大规模 → Milvus（开源，可扩展）
- 需要混合搜索（向量+关键词）→ Weaviate

### 2.4 检索策略：从简单到高级

#### 策略一：纯向量搜索

把问题和所有文档都转成向量，计算相似度，返回最相似的Top-K。

**优点**：简单、快速
**缺点**：可能漏掉关键词匹配的内容

#### 策略二：关键词搜索（BM25）

传统的全文检索，基于关键词匹配。

**优点**：精确匹配关键词效果好
**缺点**：无法理解语义（搜"忘记密码"可能找不到"重置密码"）

#### 策略三：混合搜索（Hybrid Search）

向量搜索 + 关键词搜索，加权融合。

**优点**：兼顾语义理解和关键词匹配
**缺点**：需要调权重参数

**推荐：混合搜索是RAG的最佳实践。**

#### 策略四：重排序（Rerank）

先粗筛（混合搜索返回Top-50），再精排（用专门的Rerank模型对Top-50重新排序，选Top-5）。

**优点**：精度最高
**缺点**：多一步计算，增加延迟

**推荐：对精度要求高的场景使用Rerank。**

### 2.5 查询改写：让搜索更准

用户的问题往往不是最佳的搜索查询。查询改写的目标是把用户的问题转化成更适合搜索的形式。

**用户问题：** "密码不对，进不去"

**改写后：** "用户密码错误无法登录，如何重置密码"

改写的好处：
1. 补充了隐含信息（"密码不对" = 需要重置密码）
2. 使用了更规范的术语（"进不去" → "无法登录"）
3. 提高了检索的准确率

**改写方法：**

1. **直接改写**：让大模型把用户问题改写成搜索查询
2. **多查询生成**：生成3-5个不同角度的查询，分别检索后合并结果
3. **HyDE（假设性文档嵌入）**：让模型先"假设"一个答案，用这个假设答案去检索

## 三、企业应用：RAG系统的评估

### 3.1 RAG评估的三大指标

评估一个RAG系统，需要看三个指标：

**指标一：召回率（Recall）**

> 所有相关的文档中，系统找到了多少？

举例：

- 知识库中有10篇关于"密码重置"的文档
- 用户问"怎么重置密码"
- 系统检索到了8篇
- 召回率 = 8/10 = 80%

**召回率低意味着**：系统漏掉了相关文档，模型可能得不到足够的信息来生成准确答案。

**指标二：精确率（Precision）**

> 系统检索到的文档中，有多少是真正相关的？

举例：

- 系统检索到了5篇文档
- 其中3篇真正相关，2篇不相关
- 精确率 = 3/5 = 60%

**精确率低意味着**：系统检索到了很多不相关的文档，浪费了Token，也可能误导模型。

**指标三：答案相关性（Answer Relevance）**

> 模型生成的答案，在多大程度上回答了用户的问题？

这个指标最难评估，因为它涉及到"答案好不好"的判断。

**评估方法：**

1. **人工评估**：人来看答案是否相关（最准确，但最慢）
2. **模型评估**：让另一个模型来判断答案是否相关（快，但可能有偏差）
3. **基于事实性**：检查答案中的每个事实陈述，是否都能在检索到的文档中找到依据（最客观）

### 3.2 RAG评估的RAGAS框架

RAGAS是一个流行的RAG评估框架，它定义了四个核心指标：

| 指标 | 含义 | 计算方法 |
|------|------|---------|
| 语义相似度 | 答案和标准答案的语义相似度 | 用Embedding计算 |
| 事实一致性 | 答案中的每个事实是否都能在上下文找到 | 模型判断 |
| 上下文召回率 | 检索到的上下文是否覆盖了所有必要信息 | 模型判断 |
| 上下文精确率 | 检索到的上下文是否都是必要的 | 模型判断 |

**产品经理需要做什么？**

1. 建立评估数据集：准备100+条"问题-标准答案-参考文档"的三元组
2. 运行评估：用RAGAS或其他工具跑评估
3. 分析结果：找出得分低的环节，针对性优化

### 3.3 客户工单系统的RAG搭建

让我们回到贯穿全书的案例。

**需求：** 客服AI需要根据内部操作手册回答用户问题。

**知识库内容：**

- 产品使用手册（50页）
- FAQ文档（200条常见问题）
- 操作指南（30个常见任务的步骤说明）
- 已知问题列表（100个已知bug及其解决方案）

**Step 1：文档处理**

```
原始文档（PDF/Word/Markdown）
        ↓
文本提取（OCR如果扫描件）
        ↓
文本清洗（去除页眉页脚、空行等）
        ↓
文档切分（Chunk size=500, Overlap=100）
        ↓
向量化（Embedding）
        ↓
存入向量数据库
```

**Step 2：检索策略**

```
用户问题："怎么导出报表？"
        ↓
查询改写 → "产品报表导出操作步骤"
        ↓
混合搜索（向量搜索 + BM25关键词搜索）
        ↓
Top-10结果
        ↓
Rerank模型精排
        ↓
Top-3最相关chunk
        ↓
组装Prompt → 发送给模型
```

**Step 3：Prompt组装**

```
你是一个专业的客服专家。请根据以下知识库内容回答用户问题。

【知识库】
{Top-3 chunk内容}

【用户问题】
{用户问题}

要求：
1. 只基于知识库内容回答
2. 如果知识库中没有相关信息，告知用户将转接人工
3. 回复控制在200字以内
4. 语气友好，步骤清晰
```

**Step 4：效果评估**

```
评估数据集：50条"问题-标准答案"对
评估指标：
- 召回率 ≥ 80%
- 精确率 ≥ 70%
- 答案相关性 ≥ 75%
```

## 四、产品经理需要掌握什么？

### 4.1 RAG产品经理的"最小知识集"

你不需要理解Embedding的数学细节，但需要知道：

1. **Embedding质量决定检索质量**：如果Embedding模型不能很好地理解你的领域语言，检索就不准。
2. **Chunk大小影响召回和精确的平衡**：太小召回好但精确差，太大精确好但召回差。
3. **检索不是万能的**：如果知识库本身就没有相关内容，检索再准也没用。
4. **RAG的效果是"端到端"的**：从文档质量、切分策略、Embedding模型、检索策略、到最终的Prompt设计，每一步都影响最终效果。

### 4.2 RAG产品的PRD写法

**模块一：知识库描述**

```
知识库内容：
- 产品使用手册（50页PDF）
- FAQ文档（200条）
- 操作指南（30个任务）
- 已知问题列表（100条）

知识库更新频率：每周更新
知识库负责人：产品团队
```

**模块二：检索策略**

```
检索方式：混合搜索（向量搜索60% + BM25关键词搜索40%）
Top-K：粗筛Top-10，重排序后Top-3
相似度阈值：>0.7才计入相关文档
```

**模块三：效果指标**

```
- 召回率 ≥ 80%（至少找到80%的相关文档）
- 精确率 ≥ 70%（检索结果中至少70%是相关的）
- 答案相关性 ≥ 75%（模型回答至少75%能准确回答问题）
```

**模块四：兜底策略**

```
- 检索到的文档相似度都<0.5 → 转人工
- 知识库中没有相关内容 → 告知用户并转人工
- 检索超时（>3秒） → 降级为纯模型回答
```

### 4.3 跟工程师沟通的常用话术

**讨论知识库建设时：**

## 七、真实案例：某SaaS公司的"AI知识库助手"

### 7.1 背景

某项目管理SaaS公司有5万付费用户，每天有约2000个用户咨询。其中约60%的问题是产品使用相关的问题：

- "怎么创建项目？"
- "怎么邀请团队成员？"
- "怎么设置权限？"
- "怎么导出数据？"

原来的方案：

- 一个静态FAQ页面（200个问题）
- 用户需要自己搜索关键词
- 搜索不准就放弃，直接发邮件咨询

问题：

- FAQ更新不及时（产品更新了，FAQ没更新）
- 用户搜索关键词不准确（搜"创建项目"找不到，应该搜"新建项目"）
- 邮件咨询响应慢（平均4小时）

### 7.2 解决方案

搭建RAG知识库助手：

**知识库内容：**

- 产品使用手册（50页）
- FAQ文档（200条，每周更新）
- 产品更新日志（2年历史）
- 已知问题列表（100条）

**技术架构：**

```
文档 → 解析 → 切分（500字/块，100字重叠）
  → Embedding（text-embedding-3-small）
  → 存入Pinecone
  → 用户提问 → 查询改写 → 混合搜索 → Rerank → 组装Prompt → 生成回答
```

**检索策略：**

- 混合搜索：向量搜索60% + BM25 40%
- Top-10粗筛 → Rerank → Top-3
- 相似度阈值：0.65

### 7.3 效果

上线三个月后：

- 产品使用类咨询的自动解决率：72%（原来FAQ的搜索只能解决35%）
- 用户满意度：81%（原来邮件咨询的满意度65%）
- 平均响应时间：从4小时降到3秒
- 月度客服成本降低：40%

### 7.4 关键经验

1. **知识库质量比检索技术更重要**：知识库文档质量差，再好的检索也没用。他们花了第一个月专门整理知识库，效果提升最大。
2. **查询改写带来了显著改善**：用户说"创建项目"，改写后搜"新建项目"也能找到。这解决了FAQ搜索的最大痛点。
3. **Rerank提升了精确率**：不加Rerank时，Top-3中有30%是不相关的。加了Rerank后降到8%。
4. **持续运营是关键**：知识库需要每周更新，FAQ需要每周审核，Bad Case需要每周分析。RAG不是一次性项目，是持续运营。

## 八、常见误区

### 误区一："RAG就是向量搜索"

**真相**：向量搜索只是RAG的一个环节。RAG还包括文档切分、查询改写、重排序、Prompt组装、答案生成等多个环节。向量搜索只是其中一环。

**实践方法**：全面优化RAG的每个环节，不要只关注向量搜索。

### 误区二："Embedding模型越强越好"

**真相**：Embedding模型的选择需要考虑领域适配性。一个在通用语料上训练的强Embedding模型，可能不如一个在特定领域微调过的弱模型。

**实践方法**：在目标领域上做小规模测试，选择效果最好的Embedding模型，不要只看模型排行榜。

### 误区三："Chunk越大越好"

**真相**：Chunk太大会丢失检索精度——一大块文档里可能包含多个主题，检索时无法精确定位到相关部分。Chunk太小会丢失上下文——一个步骤说明可能被切成两半。

**实践方法**：默认500字，根据文档类型调整。结构化文档（手册、指南）用较大的Chunk，非结构化文档（聊天记录、论坛帖子）用较小的Chunk。

### 误区四："检索到了相关内容，模型就一定能用好"

**真相**：检索到的内容再多，如果Prompt没有告诉模型"如何使用这些内容"，模型也可能忽略或误用。

**实践方法**：在Prompt中明确指示模型"只基于以下文档回答问题"，并给出使用文档的指导。

### 误区五："RAG的效果是静态的"

**真相**：RAG的效果会随着时间变化。知识库在更新，用户的问题在变化，模型在升级。今天的Good效果，明天可能就不行了。

**实践方法**：建立持续监控和迭代机制，至少每月评估一次RAG效果。

### 误区六："RAG可以替代一切"

**真相**：RAG解决的是"知识"问题，不是"能力"问题。如果任务需要执行操作（如查订单、发邮件），RAG不够，需要Function Calling。

**实践方法**：RAG和Function Calling配合使用——RAG获取知识，Function Calling执行操作。

### 误区七："知识库越大越好"

**真相**：知识库太大反而可能降低检索精度——更多的文档意味着更多的噪声，检索时更容易混入不相关的内容。

**实践方法**：知识库需要定期清理——删除过时文档、合并重复文档、补充缺失文档。质量 > 数量。

## 九、面试题

### 初级

**Q1：什么是RAG？它解决了什么问题？**

**参考答案：**

RAG全称Retrieval-Auganced Generation（检索增强生成）。它的核心思想是：让模型在回答问题之前，先从外部知识库中检索相关信息，再基于这些信息生成回答。

解决的问题：

1. 模型的知识有截止时间，RAG可以让模型"看到"最新的信息
2. 模型不知道你的内部数据，RAG可以让模型"读取"你的文档
3. 模型的回答无法溯源，RAG可以让回答附带来源文档

---

**Q2：RAG和微调有什么区别？**

**参考答案：**

RAG是"检索时临时获取知识"——模型在回答问题的瞬间，从知识库中检索相关信息。好处是即时更新、可以处理海量文档、可以溯源。

微调是"训练时永久学习知识"——用你的数据重新训练模型，让模型"学会"你的知识。好处是学习特定风格和格式，响应更快（不需要检索）。

简单来说：RAG是"查一下再用"，微调是"学一下再用"。

---

### 中级

**Q3：你的RAG系统召回率只有60%，怎么提升？**

**参考答案：**

召回率低意味着系统漏掉了太多相关文档。我会从以下几个方面排查：

1. **文档切分**：Chunk大小是否合适？是否按语义完整切分？尝试调整Chunk大小和重叠。

2. **Embedding模型**：当前Embedding模型是否能很好地理解你的领域语言？尝试换更强的Embedding模型或在领域数据上微调Embedding。

3. **检索策略**：只用向量搜索吗？尝试混合搜索（向量+BM25）。

4. **查询改写**：用户的问题是否太口语化？加入查询改写环节，把用户问题转成更适合搜索的形式。

5. **知识库质量**：知识库中是否有相关内容？如果知识库本身就缺少某些文档，召回率不可能高。

---

**Q4：RAG系统中，检索到的文档质量不好（内容不完整、格式混乱），怎么办？**

**参考答案：**

这是知识库预处理的问题。解决方案：

1. **文档解析优化**：确保PDF、Word等格式能正确解析为纯文本。扫描件需要OCR。

2. **文本清洗**：去除页眉页脚、空行、特殊字符。统一格式。

3. **结构化处理**：对于有明确结构的文档（如手册），保留章节层级信息。每个chunk带上章节标题作为上下文。

4. **人工审核**：对关键文档进行人工审核，确保内容准确、格式正确。

5. **建立文档规范**：制定文档编写规范，从源头保证文档质量。

---

### 高级

**Q5：你负责一个RAG系统，发现模型经常"忽略"检索到的文档，用自己的知识回答。你怎么办？**

**参考答案：**

这是RAG系统中常见的问题，叫"模型遗忘"或"文档忽略"。

**原因分析：**

1. **Prompt没有强制约束**：模型没有被明确要求"只基于检索到的文档回答"。
2. **检索到的文档质量差**：模型"看出来"了文档不相关，所以选择忽略。
3. **模型本身的训练偏好**：模型在训练时被教导"不要编造信息"，但检索到的文档信息不足时，它会倾向于用自己的知识。

**解决方案：**

1. **加强Prompt约束**：
   ```
   请只基于以下知识库内容回答用户问题。
   如果知识库中没有相关信息，请回答"抱歉，我没有找到相关信息"。
   不要使用你自己的知识回答。
   ```

2. **提高检索质量**：确保检索到的文档与用户问题高度相关。

3. **在Prompt中标注来源**：让模型知道哪些信息来自知识库。
   ```
   【知识库来源1】...
   【知识库来源2】...
   请基于以上来源回答问题。
   ```

4. **评估模型是否真的"忽略"了**：有时候模型用了知识库的信息，但回答方式不像在"引用"文档。需要具体分析Bad Case。

## 十、本章总结

本章我们系统学习了RAG的核心原理和最佳实践：

1. **RAG的本质**：让AI在回答之前先"查一下资料"，解决大模型知识的时效性和私有性问题
2. **文档切分**：Chunk大小、重叠、粒度——切分策略直接影响检索质量
3. **Embedding和向量搜索**：语义相似的文字在向量空间中距离近，这是RAG检索的基础
4. **检索策略**：纯向量搜索 → 关键词搜索 → 混合搜索 → 重排序，精度递增但延迟也递增
5. **查询改写**：把用户口语化的问题转成更适合搜索的形式
6. **评估指标**：召回率、精确率、答案相关性——三个指标缺一不可
7. **客户工单系统RAG搭建**：从文档处理到检索策略到效果评估的完整流程
8. **SaaS知识库案例**：从35%到72%的自动解决率提升

# 第2章 需求先行——什么场景真正需要AI


---

## 一、背景：为什么"要不要用AI"比"怎么用AI"更重要？

### 1.1 一个常见的悲剧故事

> 某公司的产品经理接到一个需求："用户投诉响应时间太长，平均要48小时。"
>
> 他的第一反应是："用AI啊！让AI自动回复用户投诉。"
>
> 团队花了两个月开发了AI投诉处理系统，上线后发现：AI回复的准确率低，用户更加不满，投诉量反而上升了30%。
>
> 最后不得不回退到人工处理，两个月的工时白费了。

这个故事不是虚构的。我在多家公司的产品团队中听过类似的故事。问题的根源不在于技术，而在于**需求判断的顺序错了**。

正确的顺序应该是：

```
需求是什么？ → 为什么有这个需求？ → AI适合解决这个问题吗？ → 如果适合，用什么AI技术？
```

错误的顺序是：

```
要用AI！ → 找个场景试试 → 做不出来再说
```

### 1.2 为什么很多"AI需求"其实是伪需求？

让我们从一个具体的例子开始。

假设你是一家银行的产品经理。业务方提了一个需求："我们想用AI来预测客户会不会流失。"

这个需求听起来很合理吧？但让我们深挖一层：

**业务方的真实目标是什么？** 降低客户流失率，保住存款。

**"预测流失"是达成目标的最好方式吗？** 不一定。也许更好的方式是：
- 分析客户流失的原因（利率太低？服务不好？竞品更便宜？）
- 针对流失原因设计挽留策略（提高利率、专属客服、礼品赠送）
- 在流失发生前主动联系客户

"预测流失"只是手段之一，而且是一个"事后诸葛亮"的手段——客户都已经想走了，你再预测出来，挽留率可能不到20%。

**真正的好需求是："如何降低客户流失率"，而不是"如何预测客户流失"。**

前者开放了所有可能的解决方案，后者把自己锁定在了一个未必最优的技术方案上。

### 1.3 AI需求的三个层次

我把产品需求分成三个层次：

**第一层：表面需求（Surface Request）**

用户或业务方直接说的话。

- "我想要一个AI聊天机器人"
- "我想用大模型来写周报"
- "我想做一个智能推荐"

这些是表面需求。它们有一个共同特点：**已经包含了技术方案**。

产品经理的第一项工作，就是把技术方案从需求中剥离出来，还原成真正的问题。

**第二层：真实需求（Real Need）**

剥掉技术方案后，用户真正想解决的问题。

- "我想要一个AI聊天机器人" → 真实需求："客户咨询响应太快，人工客服不够用"
- "我想用大模型来写周报" → 真实需求："员工写周报花了太多时间，影响工作效率"
- "我想做一个智能推荐" → 真实需求："用户在我们平台上找不到想要的东西，流失率高"

**第三层：最优解（Best Solution）**

针对真实需求，找到最好的解决方案。AI可能不是最优解。

- "客户咨询响应太快，人工客服不够用" → 最优解可能是：AI客服（如果问题类型简单且重复）；也可能是：优化排班系统（如果问题是高峰时段人手不足）；也可能是：自助服务页面（如果大部分问题是FAQ）
- "员工写周报花了太多时间" → 最优解可能是：AI自动生成（如果周报内容来自系统数据）；也可能是：简化周报模板（如果问题是模板太复杂）；也可能是：取消周报（如果问题是周报本身就没有价值）
- "用户在我们平台上找不到想要的东西" → 最优解可能是：AI推荐（如果用户行为数据丰富）；也可能是：改进搜索（如果用户知道自己在找什么）；也可能是：优化分类导航（如果用户不知道自己在找什么）

**核心原则：永远从真实需求出发，而不是从技术方案出发。**

## 二、核心原理：需求判断框架

### 2.1 AI适用性决策树

这是本章最重要的工具。面对任何需求，用这个决策树来判断是否适合用AI：

```
                    ┌─────────────────────┐
                    │   用户提出了一个需求   │
                    └─────────┬───────────┘
                              │
                              ▼
                    ┌─────────────────────┐
                    │ 这个需求的核心问题是   │
                    │ 信息处理吗？           │
                    │ (输入→转换→输出)       │
                    └─────────┬───────────┘
                           YES│    NO
                              ▼
                    ┌─────────────────────┐
                    │ 问题的规则能穷举吗？   │
                    │ (用if-then能覆盖所有  │
                    │  情况吗？)             │
                    └─────────┬───────────┘
                   YES        │       NO
                    │         ▼         │
                    │  ┌───────────┐    │
                    │  │ 不用AI    │    │
                    │  │ 规则引擎   │    │
                    │  │ 就够了     │    │
                    │  └───────────┘    │
                    │                   │
                    ▼                   ▼
            ┌──────────────┐   ┌──────────────┐
            │  需要理解     │   │ 需要精确计算  │
            │  模糊/自然    │   │ 或确定性逻辑  │
            │  语言吗？     │   │               │
            └──────┬───────┘   └───────────────┘
           YES─────┘    NO
                    │
                    ▼
            ┌──────────────┐
            │  有足够的数据 │
            │  用于训练/    │
            │  检索吗？     │
            └──────┬───────┘
           YES─────┘    NO
                    │
                    ▼
            ┌──────────────┐
            │  容错率足够   │
            │  吗？(出错    │
            │  后果能接受   │
            │  吗？)        │
            └──────┬───────┘
           YES─────┘    NO
                    │
                    ▼
            ┌──────────────┐
            │ ✅ 适合用AI   │
            └──────────────┘
```

让我们逐层解释这个决策树。

**第一层：是信息处理问题吗？**

所有产品本质上都是信息处理系统。但如果一个问题的核心不是"处理信息"，那AI就不是首选。

举例：
- 是信息处理："用户问'我的订单到哪了'，系统需要理解问题并从数据库查到结果返回"
- 不是信息处理："用户需要点击'支付'按钮来完成付款"——这是操作，不是信息处理

**第二层：规则能穷举吗？**

如果可以穷举，说明问题结构简单，用规则引擎就够了，不需要AI。

举例：
- 不能穷举："用户可能用各种方式问物流问题——'我的快递到哪了'、'东西发了吗'、'什么时候到'、'帮我查一下订单12345的物流'"——这些表达方式太多，无法用if-then穷举
- 可以穷举："用户选了A商品就推荐B商品，选了C商品就推荐D商品"——规则有限，可以直接写

**第三层：有足够的数据吗？**

AI需要数据。没有数据，AI就是无米之炊。

这里的数据包括两种：
- 训练数据：如果你想微调模型，需要有标注好的数据
- 知识库数据：如果你用RAG，需要有高质量的文档

举例：
- 有数据：电商平台有大量用户行为数据（浏览、点击、购买），可以用来训练推荐模型
- 没有数据：一个刚起步的垂直领域平台，几乎没有用户数据，用AI推荐就是空谈

**第四层：容错率足够吗？**

这是最关键的一层。AI的输出是概率性的，不可能100%准确。如果你的场景不能容忍错误，那就不能用纯AI方案。

举例：
- 容错率高：AI写邮件草稿——写错了用户可以改
- 容错率低：AI做手术建议——错了会出人命的

### 2.2 AI做不好的场景类型

知道"什么时候用AI"很重要，但知道"什么时候不用AI"更重要。以下是AI做不好的五类场景：

**类型一：高精度要求的计算**

AI不擅长精确计算。

举例：
- ❌ 用AI做财务报表汇总——AI可能会算错数字，而且你不知道它什么时候会错
- ✅ 用传统程序做财务报表汇总——100%准确

**类型二：实时性要求极高的场景**

大模型的响应延迟通常在几百毫秒到几秒之间。如果场景要求毫秒级响应，AI不合适。

举例：
- ❌ 用AI做高频交易决策——等AI想完，行情早就变了
- ✅ 用传统算法做高频交易——微秒级响应

**类型三：强合规要求的场景**

某些行业（金融、医疗、法律）有严格的合规要求，需要可追溯、可审计。AI的"概率性"输出很难满足这些要求。

举例：
- ❌ 用AI自动生成法律判决书——法官的判决需要引用具体法条和判例，AI可能编造法条
- ✅ 用AI辅助律师查找相关判例——律师自己判断，最终责任在人

**类型四：数据极度稀缺的领域**

AI需要数据来学习和理解。如果一个领域几乎没有数据，AI就无从下手。

举例：
- ❌ 用AI预测一种从未被研究过的新型病毒——没有历史数据，模型学不到规律
- ✅ 用AI分析已有大量病例的常见疾病——数据充足，模型可以学到规律

**类型五：需要深度情感连接的场景**

AI可以模拟情感，但不能真正理解情感。在某些需要深度信任和情感连接的场景，AI的参与反而会降低用户体验。

举例：
- ❌ 用AI完全替代心理咨询师——来访者需要的是真正的人的理解和共情
- ✅ 用AI做心理咨询的辅助工具——比如情绪记录、预约提醒、简单的正念引导

### 2.3 需求分析的"五个为什么"

当业务方提出一个需求时，不要急着判断"要不要用AI"，先用"五个为什么"挖到真实需求。

这个方法来自丰田的精益生产理念，但在AI场景下特别有用。

**案例演示：**

业务方说："我们需要一个AI功能，自动给用户发个性化营销消息。"

**第一个为什么：** 为什么要自动发消息？
→ 业务方答："为了提高用户活跃度。"

**第二个为什么：** 为什么要提高活跃度？
→ 业务方答："因为上个月活跃度下降了15%。"

**第三个为什么：** 活跃度为什么下降？
→ 业务方答："因为我们上一个版本改版后，新用户找不到核心功能。"

**第四个为什么：** 那为什么不用AI，先解决改版问题？
→ 业务方答："改版要两个月才能完成，但活跃度下降的压力这周就要缓解。"

**第五个为什么：** 所以真实需求是"在不改动产品的前提下，快速提升新用户活跃度"？
→ 业务方答："对。"

**结论：**

表面需求是"AI发营销消息"，但真实需求是"快速提升新用户活跃度"。

AI发营销消息可能是一个有效的短期方案，但不是唯一的方案。其他方案包括：
- 在新用户引导流程上加一个"热门功能"提示
- 给新用户发一个限时优惠
- 在首页加一个"新手专区"

**产品经理的价值就在于此：不只接受表面需求，而是挖掘真实需求，然后给出最优的解决方案组合。**

### 2.4 AI场景优先级评分卡

当你判断"这个场景适合用AI"之后，还需要判断"这个场景值不值得现在做"。

我设计了一个三维评分卡，从以下三个维度给每个AI场景打分：

**维度一：影响力（Impact）**

这个AI功能对用户和业务的价值有多大？

- 5分：核心业务流程，影响大量用户
- 3分：次要流程，影响部分用户
- 1分：边缘功能，影响少量用户

**维度二：技术可行性（Feasibility）**

用当前的AI技术，实现这个功能的把握有多大？

- 5分：技术成熟，有大量成功案例，数据充足
- 3分：技术上可行但有风险，数据或能力有缺口
- 1分：技术不成熟，缺乏数据，成功率低

**维度三：数据成熟度（Data Readiness）**

你有没有足够的数据来支撑这个AI功能？

- 5分：有高质量的数据，已经整理好
- 3分：有数据但需要清洗和整理
- 1分：几乎没有数据，需要从头积累

**评分卡示例：客户工单系统的三个AI功能**

| 功能 | 影响力 | 技术可行性 | 数据成熟度 | 总分 | 优先级 |
|------|--------|-----------|-----------|------|--------|
| 智能工单分类 | 5 | 5 | 4 | 14 | P0（最高） |
| 自动回复常见问题 | 4 | 4 | 3 | 11 | P1 |
| 工单情绪预警 | 3 | 2 | 1 | 6 | P3（最低） |

**解读：**

- 智能工单分类：影响力大（所有工单都要分类）、技术成熟（分类是NLP的经典任务）、数据充足（历史工单标签齐全）→ 优先级最高
- 自动回复常见问题：影响力较大（减少人工工作量）、技术可行但有挑战（需要结合知识库）、数据一般（FAQ文档需要整理）→ 优先级中等
- 工单情绪预警：影响力一般（只在极端情况下有用）、技术不成熟（情绪理解容易误判）、数据不足（没有标注的情绪数据）→ 优先级最低

## 三、企业应用：客户工单智能系统案例

### 3.1 案例背景

这是贯穿全书的企业案例。从这一章开始，我们会反复用到这个案例，逐步用前面学到的知识来设计和实现它。

**公司背景：**

某中型SaaS公司，产品是一款项目管理工具（类似Teambition或飞书项目）。公司有5万付费企业用户，日均创建工单量约3000条。

**现状问题：**

1. 工单分类依赖人工：客服看完工单内容，手动选择分类（技术故障/功能建议/账号问题/其他）
2. 响应速度慢：平均响应时间4小时，用户满意度72%
3. 人力成本高：需要20人客服团队处理3000条/天的工单
4. 知识传递困难：新员工需要2周才能熟练掌握分类标准和回复话术

**业务目标：**

- 将平均响应时间缩短到30分钟以内
- 将人工客服工作量减少50%
- 将用户满意度提升到85%以上

### 3.2 需求判断

用第二章的决策树来判断：

**第一层：是信息处理问题吗？**

是的。工单处理的核心是：
- 输入：用户的问题描述
- 处理：理解问题、分类、查找解决方案、生成回复
- 输出：分类结果 + 回复内容

**第二层：规则能穷举吗？**

不能。用户的工单描述千奇百怪：
- "登录后页面一直是加载状态"
- "你们的导出功能怎么用？"
- "我想把团队从10人升级到50人"
- "bug！！！"

用if-then规则穷举是不可能的。

**第三层：有足够的数据吗？**

有。公司积累了两年的工单数据，包括：
- 工单内容（用户描述）
- 工单分类标签（人工标注）
- 工单回复记录（客服回复）
- 工单解决状态（已解决/未解决）

约20万条历史工单数据，足够训练一个分类模型。

**第四层：容错率足够吗？**

较高。工单分类错误不会造成严重后果：
- 分类错了，可以转人工复核
- 回复不准确，客服可以修改后再发送

**结论：✅ 适合用AI。**

### 3.3 场景优先级评估

我们列出三个候选AI功能，用评分卡评估：

| 功能 | 影响力 | 技术可行性 | 数据成熟度 | 总分 | 优先级 |
|------|--------|-----------|-----------|------|--------|
| 工单自动分类 | 5 | 5 | 5 | 15 | P0 |
| 智能回复建议 | 4 | 4 | 3 | 11 | P1 |
| 工单紧急度自动判断 | 3 | 3 | 4 | 10 | P1 |

**P0：工单自动分类**

这是优先级最高的功能，原因：
1. 分类是所有工单处理的第一步，影响后续所有环节
2. 分类是NLP的经典任务，技术非常成熟
3. 有20万条带标签的历史数据

**P1：智能回复建议**

这个功能的价值在于减少客服打字时间，但：
1. 技术难度比分类高（需要结合知识库生成回复）
2. 数据质量一般（历史回复质量参差不齐）
3. 容错率较低（回复不准确会影响用户体验）

**P1：工单紧急度自动判断**

这个功能的价值在于优先处理紧急工单，但：
1. 紧急度的定义本身就模糊（什么是"紧急"？）
2. 需要用户反馈来校准判断准确性

### 3.4 最终需求定义

经过以上分析，我们确定了首期AI改造的范围：

**第一期（本期）：工单自动分类**

- 目标：将工单自动分类，准确率≥85%
- 流程：用户提交工单 → AI自动分类 → 路由到对应团队 → 人工客服处理
- 兜底：分类置信度<70%的工单，人工复核

**第二期（下期）：智能回复建议**

- 目标：为客服提供回复建议，减少打字时间
- 流程：客服查看工单 → AI根据工单内容推荐回复 → 客服选择/修改 → 发送给用户
- 兜底：客服可以不使用AI建议，手动编写

**第三期（远期）：工单紧急度判断**

- 目标：自动识别紧急工单，优先处理
- 流程：AI判断紧急度 → 高紧急度工单置顶 → 优先分配
- 兜底：人工可以随时调整紧急度

## 四、产品经理需要掌握什么？

### 4.1 在PRD中描述"不确定性"

这是AI产品经理最重要的技能之一。传统PRD描述的是确定性行为，AI产品的PRD需要描述概率性行为。

**传统PRD写法（确定性）：**

> 用户点击"搜索"按钮后，系统应在1秒内返回包含关键词的商品列表。如果结果为空，显示"没有找到相关商品"。

**AI产品PRD写法（概率性）：**

> 用户输入自然语言问题后，系统在2秒内返回AI回复。回复的准确率目标为85%。当回复置信度低于70%时，显示"我正在学习中，您的回答对我们很重要"并转人工客服。

注意区别：

- 传统PRD用的是"应""必须"——二元判断
- AI PRD用的是"目标为""当...时"——概率判断 + 兜底机制

### 4.2 AI产品PRD的核心模块

AI产品的PRD需要包含以下传统PRD没有的模块：

**模块一：能力描述**

> 这个AI功能"能做什么"和"不能做什么"。

示例：

> AI工单分类功能：
> - 能做：将工单分类为技术故障/功能建议/账号问题/其他四类
> - 不能做：识别不在上述四类中的新型问题类型
> - 不能做：理解纯图片工单（不含文字描述）

**模块二：效果指标**

> 这个AI功能"做得好"的标准是什么。

示例：

> - 总体分类准确率≥85%
> - 每类问题的召回率≥80%
> - 95%的请求在3秒内响应

**模块三：Bad Case处理**

> 当AI"做错了"时，系统怎么处理。

示例：

> - 置信度≥85%：自动分类，直接路由
> - 70%≤置信度<85%：自动分类，但标记为"待确认"
> - 置信度<70%：转人工分类

**模块四：数据反馈机制**

> 如何收集用户使用数据来持续优化AI。

示例：

> - 用户在AI回复后点击"有帮助"/"没有帮助"
> - 客服修改AI建议的分类时，记录修改前后的差异
> - 每周生成Bad Case报告，纳入下一轮优化

### 4.3 技术沟通：和工程师说"同一种语言"

传统PM和工程师沟通的方式：

> PM："用户点击按钮后，跳转到搜索页面。"
> 工程师："好。"

AI PM和工程师沟通的方式：

> PM："用户输入自然语言问题后，系统需要理解意图并返回相关文档。我期望准确率在85%以上，低于70%置信度的转人工。"
> 工程师："85%的准确率用现成的RAG方案可以做到。70%的阈值需要根据实际数据来调。你有多少历史数据？"
> PM："大约20万条带标签的工单记录。"
> 工程师："够了。我们可以先用这些做分类模型，然后再做RAG。"

看到了吗？第二种沟通方式中，PM用了工程师听得懂的语言（准确率、置信度、历史数据），工程师就能立刻给出技术方案。

**关键术语速查表：**

| 术语 | 含义 | 什么时候用 |
|------|------|-----------|
| 准确率（Precision） | AI说"是"的时候，有多少是真的"是" | 评估AI判断的可靠性 |
| 召回率（Recall） | 所有真的"是"里面，AI找到了多少 | 评估AI会不会漏掉 |
| 置信度（Confidence） | AI对自己判断的把握程度（0-100%） | 设置自动/人工的分界线 |
| Bad Case | AI判断错误的案例 | 分析和优化AI效果 |
| 微调（Fine-tuning） | 用特定数据训练模型 | 通用模型不够用时 |
| RAG | 检索增强生成 | 需要模型"记住"你的数据时 |

## 五、研发如何实现：AI需求的工程流程

### 5.1 从需求到技术方案的转化

当产品经理判断"这个场景适合用AI"后，研发流程大致如下：

```
产品经理输出
    │
    ├── 需求文档（PRD）
    │     ├── 能力描述（做什么/不做什么）
    │     ├── 效果指标（准确率≥85%）
    │     ├── Bad Case处理（置信度<70%转人工）
    │     └── 数据反馈机制（用户标记"没有帮助"）
    │
    ▼
技术团队评估
    │
    ├── 数据评估：有多少数据？质量如何？
    ├── 技术选型：用什么模型？RAG还是微调？
    ├── 可行性验证：POC（概念验证）测试
    ├── 方案设计：系统架构 + 接口定义
    └── 排期评估：需要多长时间？
    │
    ▼
开发 + 迭代
    │
    ├── 数据准备（清洗、标注）
    ├── 模型训练/配置
    ├── Prompt设计
    ├── 系统集成
    ├── 效果测试
    └── 迭代优化（通常2-3轮）
    │
    ▼
上线 + 监控
    │
    ├── 灰度发布（先10%用户）
    ├── 效果监控
    ├── 收集反馈
    └── 持续优化
```

### 5.2 产品经理在研发流程中的关键参与点

产品经理不需要参与每一行代码的编写，但需要在以下节点深度参与：

**节点一：需求评审**

- 输出：完整的PRD（含能力描述、效果指标、Bad Case处理）
- 目标：确保技术团队理解"为什么要做"和"做到什么程度算好"

**节点二：POC验证**

- 参与：看技术团队的概念验证结果
- 目标：确认技术可行性，调整效果预期

**节点三：效果评审**

- 参与：测试AI输出的质量，判断是否达到PRD中的效果指标
- 目标：决定是否可以进入下一阶段

**节点四：灰度上线评审**

- 参与：查看灰度数据，判断是否可以扩大范围
- 目标：控制风险，避免大面积效果问题

## 七、真实案例：某内容平台的"AI选题推荐"

### 7.1 背景

某内容创作平台（类似知乎或Medium），有10万创作者每天发布文章。平台有一个"编辑推荐"功能，运营人员每天手动挑选优质文章推荐给全站用户。

问题：

- 运营团队只有5个人，每天能审读的文章不超过200篇
- 平台每天新增文章约5000篇
- 编辑的主观判断导致推荐风格不统一

### 7.2 需求判断

**表面需求：** "我们需要一个AI来帮我们自动推荐好文章。"

**真实需求：** "我们想让优质内容获得更多曝光，提升用户阅读时长和创作者活跃度。"

**决策树判断：**

1. 是信息处理问题吗？→ 是（理解文章内容并判断质量）
2. 规则能穷举吗？→ 不能（"好文章"的标准很难用规则穷举）
3. 有足够数据吗？→ 有（历史推荐数据：哪些文章被推荐了、获得了多少阅读和点赞）
4. 容错率足够吗？→ 是（推荐错了只是少曝光一篇文章，不会造成负面影响）

**结论：✅ 适合用AI。**

**优先级评分：**

| 功能 | 影响力 | 技术可行性 | 数据成熟度 | 总分 |
|------|--------|-----------|-----------|------|
| AI自动推荐 | 5 | 3 | 4 | 12 |

技术可行性打3分的原因：文章质量的判断比工单分类复杂得多，需要理解文章的深度、可读性、原创性等维度。

### 7.3 产品设计

最终方案不是"AI完全替代编辑"，而是"AI辅助编辑"：

```
每日新增5000篇文章
        ↓
AI初步评分（基于阅读量预测、内容质量、原创性）
        ↓
选出Top 500篇（AI认为可能优质的）
        ↓
运营人员审核Top 500篇
        ↓
选出Top 50篇（最终推荐）
        ↓
全站曝光
```

关键点：

- AI不是直接推荐50篇，而是把筛选范围从5000缩小到500
- 运营人员的工作量从"从5000篇里找50篇"变成了"从500篇里挑50篇"
- 效率提升了10倍，同时保留了人工判断的质量

### 7.4 效果

上线三个月后：

- 运营人员的筛选效率提升8倍（每天处理文章从200篇增加到1600篇）
- 推荐文章的平均阅读量提升40%（AI发现了运营人员可能忽略的优质内容）
- 创作者活跃度提升25%（更多中小创作者的文章获得了曝光）

### 7.5 反思

这个案例最重要的经验是：**AI不是替代人，而是放大人的判断力。**

如果一开始就做"AI全自动推荐"，效果可能反而不好——因为缺少了人工判断的"最后一道防线"。正确的做法是找到AI和人的最佳配合点：AI做"粗筛"，人做"精挑"。

## 八、常见误区

### 误区一："所有需求都可以用AI做"

**真相**：AI有明确的能力边界。高精度计算、实时决策、强合规场景、数据稀缺领域、深度情感连接——这些都不适合纯AI方案。

**实践方法**：用决策树逐一排查，不要跳过任何一层。

### 误区二："AI效果好就等于产品好"

**真相**：AI的效果只是产品的一部分。即使AI准确率95%，如果用户不知道怎么用、UI设计反人类、响应速度慢，产品依然是失败的。

**实践方法**：把AI效果指标和用户体验指标分开看，两者都要达标。

### 误区三："先做AI功能，再想用在什么场景"

**真相**：这是最常见的错误。手里有个锤子，看什么都像钉子。先有技术再找场景的产品，90%以上失败了。

**实践方法**：永远从真实需求出发，用"五个为什么"挖到核心问题。

### 误区四:"效果不好就换更强的模型"

**真相**：AI产品的效果差，很多时候不是模型的问题，而是数据质量差、Prompt设计不好、或者产品流程设计不合理。

**实践方法**：建立Bad Case分析流程，先分析根因再决定优化方向。

### 误区五："AI产品上线就结束了"

**真相**：AI产品上线只是开始。数据在变化，用户行为在变化，模型效果会逐渐衰减。需要持续监控和优化。

**实践方法**：建立效果监控看板、Bad Case收集机制、每周迭代节奏。

### 误区六："准确率越高越好"

**真相**：不一定。追求99%的准确率可能需要投入10倍的成本，但业务收益可能只增加了1%。需要找到"性价比最优"的点，而不是"理论最高"的点。

**实践方法**：在PRD中定义"足够好"的标准，而不是"最好"的标准。

### 误区七："AI功能应该一步到位"

**真相**：AI产品适合MVP（最小可行产品）思路。先做一个简单版本，收集数据和反馈，再逐步迭代。

**实践方法**：第一期只做一件事（比如工单分类），做到足够好之后再扩展。

## 九、面试题

### 初级

**Q1：用户说"我想要一个AI功能"，你怎么判断这个需求是否合理？**

**参考答案：**

我会分三步来判断：

第一步：用"五个为什么"找到真实需求。用户说要AI功能，背后真正想解决的问题是什么？可能是效率问题、体验问题、或者成本问题。

第二步：用决策树判断是否适合AI。
- 是信息处理问题吗？
- 规则能穷举吗？
- 有足够数据吗？
- 容错率足够吗？

第三步：用优先级评分卡评估值不值得做。从影响力、技术可行性、数据成熟度三个维度打分。

如果三步都通过了，这个需求就是合理的。

---

**Q2：什么场景不适合用AI？举三个例子。**

**参考答案：**

1. 高精度计算场景：如财务结算、银行转账。这些场景需要100%准确，AI的概率性输出无法满足。

2. 实时性要求极高的场景：如高频交易、自动驾驶紧急制动。AI的响应延迟无法满足毫秒级要求。

3. 数据稀缺的领域：如新兴行业的专业问题。AI需要数据来训练和理解，没有数据就没法用。

---

### 中级

**Q3：你如何在一个PRD中描述一个AI功能的效果预期？**

**参考答案：**

我会从四个方面描述：

1. **能力描述**：明确AI功能"能做什么"和"不能做什么"。比如"AI能分类四种类型的工单，但不能处理图片类工单"。

2. **效果指标**：定义具体的量化标准。比如"总体分类准确率≥85%，每类召回率≥80%"。

3. **分级处理策略**：根据不同置信度定义不同的处理流程。比如"置信度≥85%自动处理，70%-85%标记待确认，<70%转人工"。

4. **反馈优化机制**：定义如何收集数据来持续优化。比如"用户标记'没有帮助'时自动归档Bad Case"。

---

**Q4：一个AI功能的准确率只有70%，但业务方要求做到95%。你怎么办？**

**参考答案：**

首先，我不会直接说"做不到"或"努力做"，而是先搞清楚几个问题：

1. **95%的标准是怎么来的？** 是业务方的直觉，还是有数据支撑？
2. **70%的准确率在实际使用中影响多大？** 如果有兜底机制（如人工审核），70%可能已经够用了。
3. **提升到95%需要什么代价？** 可能需要更多数据、更长的训练时间、更强的模型，这些成本和收益是否匹配？

然后我会给出方案：

- 如果70%的准确率配合人工审核后能达到业务目标，那就先上线70%的版本，后续迭代
- 如果必须95%，那需要评估数据量和模型选型是否支持，给出一个可行的时间表
- 如果确实做不到95%，那需要和业务方沟通，看是否能调整目标或换个方案

---

### 高级

**Q5：你负责一个AI产品，上线一个月后用户满意度从72%降到了65%。你如何分析和应对？**

**参考答案：**

我会按以下步骤处理：

**第一步：快速止血**

立即查看最近一周的Bad Case，看是否有集中出现的问题。如果发现了系统性问题（如某个分类的准确率突然暴跌），先回退到上一版本或增加人工审核比例。

**第二步：数据分析**

1. 按时间维度分析：满意度下降是从哪一天开始的？那天有什么变更？（新模型上线？新Prompt？新数据？）
2. 按问题类型分析：是哪类问题的满意度下降最多？
3. 按用户维度分析：是新用户还是老用户？是特定行业还是全行业？

**第三步：根因定位**

可能的原因包括：
- 模型漂移：用户的行为模式变了，模型没有跟上
- 数据污染：新入库的知识文档质量差，影响了RAG效果
- Prompt退化：最近的Prompt迭代引入了新问题
- 竞品影响：竞争对手推出了更好的功能，拉高了用户预期

**第四步：制定恢复计划**

根据根因制定针对性的恢复方案，设定恢复目标和时间表，定期向业务方同步进展。

**Q6：公司有两个AI需求，资源只够做一个。你会怎么选？**

需求A：智能客服（预计提升用户满意度15%，覆盖80%的用户）
需求B：智能报表生成（预计提升内部效率20%，覆盖20%的用户）

**参考答案：**

我不会只看表面数据，而是从以下几个维度综合评估：

**1. 战略匹配度**

这两个需求哪个更符合公司当前的战略目标？如果公司今年的战略是"提升用户体验"，那A优先。如果是"提升内部效率"，那B优先。

**2. 用户影响范围**

A覆盖80%的用户，B覆盖20%的用户。从影响人数看，A更大。但还要看这20%的内部用户是谁——如果是关键客户或高层决策者，影响可能更大。

**3. 技术风险和投入**

智能客服的技术相对成熟（NLP分类+RAG），风险较低。智能报表生成需要理解结构化数据并生成图表，技术难度更高，风险更大。

**4. 长期价值**

智能客服是一次性投入持续受益。智能报表生成可能需要针对不同行业定制，扩展性较差。

**综合判断**：如果两个需求战略匹配度相当，我会选A。原因：用户覆盖面广、技术风险低、长期价值大。但如果B对应的是一个战略级大客户，那B优先。

## 十、本章总结

本章我们建立了AI产品经理最重要的思维框架：**需求先行，技术后置。**

核心要点：

1. **从真实需求出发**：不要被表面的技术方案绑架，用"五个为什么"挖到真正的问题
2. **决策树判断**：四个层层递进的问题，判断是否适合用AI
3. **AI做不好的五类场景**：高精度计算、超高实时性、强合规、数据稀缺、深度情感连接
4. **优先级评分卡**：影响力、技术可行性、数据成熟度三个维度
5. **AI产品PRD的特殊性**：能力描述、效果指标、Bad Case处理、数据反馈机制
6. **企业案例启动**：客户工单智能系统，确定了第一期做"工单自动分类"

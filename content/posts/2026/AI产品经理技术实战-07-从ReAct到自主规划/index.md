## 一、背景：从"工具"到"助手"

### 1.1 什么是Agent？

Agent（智能体）是一个能**自主感知、思考、行动**的系统。

类比：

- **传统软件**：像一个计算器。你输入1+1，它输出2。你让它算什么，它算什么。它不会自己决定算什么。
- **AI Agent**：像一个助理。你说"帮我分析一下这个季度的销售"，它会自己决定：先查数据→再分析→找出问题→给出建议。

**关键区别：Agent能自主决定"怎么做"，而不只是"做什么"。**

### 1.2 Agent vs 传统AI功能

| 维度 | 传统AI功能 | Agent |
|------|-----------|-------|
| 交互方式 | 一问一答 | 多步推理和执行 |
| 自主性 | 被动响应 | 主动规划和执行 |
| 记忆 | 无记忆或短期记忆 | 长期记忆+短期记忆 |
| 工具使用 | 单次工具调用 | 多次工具调用，可能串行也可能并行 |
| 适用场景 | 简单任务（分类、翻译、摘要） | 复杂任务（分析、规划、创作） |

### 1.3 为什么Agent是AI产品的未来？

因为**用户的期望在变化**。

- 2020年之前：用户期望AI"回答问题"
- 2023年：用户期望AI"生成内容"
- 2025年：用户期望AI"完成任务"

从"回答"到"生成"到"完成"，这是产品期望的三次跃迁。Agent是实现"完成任务"这一期望的核心架构。

## 二、核心原理：Agent的构成

### 2.1 Agent的四件套模型

一个Agent由四个核心组件构成：

```
┌─────────────────────────────────────────┐
│              Agent                       │
│                                          │
│  ┌──────────┐  ┌──────────┐             │
│  │  大脑     │  │  记忆     │             │
│  │ (模型)    │  │(Memory)  │             │
│  └────┬─────┘  └────┬─────┘             │
│       │              │                   │
│  ┌────▼─────┐  ┌────▼─────┐             │
│  │  工具     │  │  行动     │             │
│  │(Tools)   │  │(Action)  │             │
│  └──────────┘  └──────────┘             │
│                                          │
│  大脑思考 → 记忆回忆 → 工具执行 → 行动   │
└─────────────────────────────────────────┘
```

**大脑（Brain）：** 大语言模型。负责推理、规划、决策。

**记忆（Memory）：** 存储历史信息。包括：
- 短期记忆（对话历史）：刚才说了什么
- 长期记忆（向量存储）：以前学过什么

**工具（Tools）：** 外部能力。包括：
- Function Calling（查数据库、发邮件）
- RAG（查知识库）
- API调用（查天气、查股价）

**行动（Action）：** 执行工具后的结果反馈给用户或下一个思考步骤。

### 2.2 ReAct范式：为什么"边想边做"最好

#### ReAct是什么？

ReAct = **Re**asoning（推理）+ **Act**ion（行动）。

核心思想：让Agent在思考和行动之间交替进行。

```
Thought: 用户想查订单12345的物流
Action: 调用get_order_info(order_id="12345")
Observation: 订单信息：tracking_id=SF123, status=shipped
Thought: 现在有了物流单号，可以查物流了
Action: 调用get_tracking(tracking_id="SF123")
Observation: 物流信息：位置=北京，ETA=明天
Thought: 现在我有了完整信息，可以回复用户了
Action: 生成回复
```

#### 为什么ReAct比"全想再做"或"全做不想"好？

**方案一：全想再做（Plan-and-Execute）**

```
Thought: 用户想查物流，我需要先查订单再查物流
Plan:
  1. 调用get_order_info
  2. 调用get_tracking
  3. 生成回复
Action: 执行计划
```

问题：计划是基于"猜测"的。如果计划错了（比如订单信息查询返回了意外结果），整个计划就废了。

**方案二：全做不想（Rule-based）**

```
Action: 调用get_order_info
Action: 调用get_tracking
Action: 生成回复
```

问题：没有推理能力。如果某个步骤失败了，不知道怎么办。

**方案三：ReAct（边想边做）**

```
Thought → Action → Observation → Thought → Action → ...
```

优势：
1. **灵活**：每一步都基于上一步的观察来决策
2. **可追溯**：每一步的Thought都是推理记录
3. **容错**：某一步失败了，Thought可以决定换一条路

#### ReAct的核心循环

```
┌─────────────────────────────────────────┐
│           ReAct 循环                     │
│                                          │
│  ┌─────────┐                            │
│  │ Thought  │ ← 推理：我现在知道了什么？  │
│  └────┬────┘    下一步该做什么？          │
│       │                                │
│       ▼                                │
│  ┌─────────┐                            │
│  │  Action  │ ← 行动：调用工具           │
│  └────┬────┘                            │
│       │                                │
│       ▼                                │
│  ┌─────────┐                            │
│  │Observation│ ← 观察：工具返回了什么？   │
│  └────┬────┘                            │
│       │                                │
│       ▼                                │
│  ┌─────────┐                            │
│  │ 回到    │ ← 基于新信息，继续推理      │
│  │ Thought  │                            │
│  └─────────┘                            │
│                                          │
│  循环直到：任务完成 或 达到最大步数       │
└─────────────────────────────────────────┘
```

### 2.3 Plan-and-Solve：复杂任务的路径规划

#### Plan-and-Solve是什么？

让Agent在开始执行之前，先"想一个计划"，然后把计划分解成步骤，逐步执行。

```
用户："帮我分析上季度销售数据，找出问题最大的三个区域，给改进建议"

Step 1（Plan）：
  1. 查询上季度销售数据
  2. 按区域汇总
  3. 找出下降最多的三个区域
  4. 分析每个区域下降的原因
  5. 生成改进建议

Step 2（Execute）：
  执行计划中的每一步
```

#### Plan-and-Solve的变体

**Self-Plannning（自我规划）：**

让Agent自己生成计划，而不是由产品经理预设。

```
系统提示词：
你是一个数据分析助手。在回答复杂问题之前，请先制定一个计划，
然后逐步执行。

用户：帮我分析上季度销售数据...

模型：
计划：
1. 查询销售数据
2. 按区域汇总
3. 找出下降最多的三个区域
4. 分析原因
5. 生成建议

执行：
步骤1：调用get_sales_data(...)
...
```

**Tree of Thoughts（思维树）：**

对于特别复杂的问题，不只生成一个计划，而是生成多个可能的计划，然后选择最好的那个执行。

```
用户：如何提升用户留存率？

可能的计划A：
1. 分析流失用户特征
2. 找出流失主要原因
3. 设计挽留策略

可能的计划B：
1. 分析高留存用户特征
2. 找出留存关键因素
3. 优化新用户引导流程

评估：计划A更直接 → 执行计划A
```

### 2.4 Reflection模式：让Agent自我检查

#### Reflection是什么？

Reflection（反思）是让Agent在完成一个任务后，回头检查一下自己的输出，看看有没有问题。

```
用户：总结一下这份财报

第一轮（生成）：
  "这份财报显示公司Q3营收增长15%，净利润下降5%..."

第二轮（反思）：
  "等等，营收增长15%但净利润下降5%？
   我需要确认一下原因。
   重新检查数据...
   哦，是因为Q3有一次性大额支出。
   修正后的总结：...'

第三轮（验证）：
  "我的总结是否包含了所有关键数据？
   营收增长率、净利润率、每股收益...
   都有了。可以回复。"
```

#### Reflection的价值

1. **减少幻觉**：模型在反思时会发现自己"编造"了数据
2. **提高质量**：反思后可以改进输出的结构和完整性
3. **自我修正**：发现错误后主动修正，而不是等用户指出

#### Reflection的实现方式

**方式一：内置Reflection Prompt**

```
请完成以下任务。完成后，请检查你的回答：
1. 是否包含了所有必要信息？
2. 是否有数据错误？
3. 是否有更好的表达方式？

如果发现问题，请修正后再回复。
```

**方式二：多Agent协作**

```
Agent A：负责生成回答
Agent B：负责审查回答
Agent C：负责最终确认
```

**方式三：工具辅助**

```
Agent生成回答后，调用一个"验证工具"来检查回答的准确性。
```

### 2.5 状态管理：Agent的记忆系统

Agent需要记忆才能"连贯地"完成任务。记忆分为三个层次：

**短期记忆（Short-term Memory）：**

- 当前对话的历史记录
- 通常限制在最近的N轮对话
- 受上下文窗口限制

**工作记忆（Working Memory）：**

- Agent在执行任务过程中"记住"的中间结果
- 比如：已经调用了哪些工具、得到了什么结果、下一步计划做什么
- 通常存储在Agent的内部状态中

**长期记忆（Long-term Memory）：**

- 跨会话的记忆
- 比如：用户的历史偏好、之前对话的关键信息
- 通常存储在向量数据库或关系数据库中

```
┌─────────────────────────────────────┐
│            Agent记忆系统              │
│                                      │
│  ┌──────────────┐                    │
│  │ 长期记忆      │ ← 用户偏好、历史   │
│  │ (Vector DB)  │   对话关键信息      │
│  └──────┬───────┘                    │
│         │ 检索相关记忆                │
│  ┌──────▼───────┐                    │
│  │ 工作记忆      │ ← 当前任务的中间   │
│  │ (In-Memory)  │   结果和状态        │
│  └──────┬───────┘                    │
│         │ 传递给当前轮次              │
│  ┌──────▼───────┐                    │
│  │ 短期记忆      │ ← 当前对话历史     │
│  │ (Context)    │                    │
│  └──────────────┘                    │
└─────────────────────────────────────┘
```

## 三、企业应用：Agent架构设计

### 3.1 Agent架构的选型

不是所有场景都需要完整的Agent架构。根据任务复杂度选择：

| 复杂度 | 架构 | 适用场景 | 示例 |
|--------|------|---------|------|
| 低 | 单步Prompt | 简单问答、翻译、摘要 | "把这段话翻译成英文" |
| 低-中 | Prompt + Function Calling | 需要调用工具的简单任务 | "查一下订单12345的状态" |
| 中 | Prompt + Function Calling + RAG | 需要知识和工具的中等任务 | "根据知识库回答用户问题" |
| 中-高 | ReAct Agent | 需要多步推理和工具调用的任务 | "分析数据并给出建议" |
| 高 | Multi-Agent | 需要多个Agent协作的复杂任务 | "完整的智能客服系统" |

### 3.2 Agent的终止条件

Agent需要知道"什么时候停下来"。否则它会无限循环。

**终止条件的设计：**

1. **任务完成**：Agent判断目标已达成
2. **达到最大步数**：防止无限循环（通常设5-10步）
3. **超时**：执行时间超过阈值（通常30秒）
4. **人工干预**：用户说"停"或Agent请求人工确认
5. **置信度阈值**：Agent对自己的判断置信度低于阈值时转人工

### 3.3 确定性 vs 随机性：Agent的控制级别

Agent的自主程度需要在"完全控制"和"完全自由"之间找到平衡。

**高控制模式：**

```
预设工作流：
用户提问 → 分类 → 检索 → 生成回复 → 审核 → 发送
```

- 优点：行为可预测，适合高合规场景
- 缺点：灵活性差，无法处理预设流程外的情况

**低控制模式：**

```
自由Agent：
用户提问 → Agent自主决定怎么做 → 执行 → 回复
```

- 优点：灵活，能处理各种情况
- 缺点：行为不可预测，适合容错率高的场景

**混合模式（推荐）：**

```
用户提问 → Agent自主规划 → 关键步骤人工确认 → 执行 → 回复
```

- 优点：灵活性和可控性的平衡
- 关键步骤（如退款、删除数据）需要人工确认

### 3.4 Agent的调试

调试Agent比调试传统软件难，因为Agent的行为是概率性的。

**调试方法：**

1. **Trace日志**：记录Agent的每一步Thought、Action、Observation
2. **Step可视化**：在UI上展示Agent的思考过程
3. **断言插入**：在关键步骤检查条件是否满足
4. **回放测试**：用相同的输入反复测试，观察行为一致性
5. **沙箱环境**：在隔离环境中测试Agent，不影响生产数据

## 四、产品经理需要掌握什么？

### 4.1 Agent设计的产品思维

设计Agent产品时，产品经理需要回答以下问题：

**问题一：这个任务需要Agent吗？**

- 如果任务简单（一问一答），不需要Agent
- 如果任务需要多步推理和执行，需要Agent

**问题二：Agent的自主程度是多少？**

- 完全自主：Agent自己决定怎么做（适合低风险场景）
- 半自主：关键步骤需要人工确认（适合中等风险场景）
- 完全受控：按预设工作流执行（适合高风险场景）

**问题三：Agent的记忆范围是多少？**

- 只记住当前对话？
- 记住历史对话？
- 记住用户偏好？

**问题四：Agent的终止条件是什么？**

- 任务完成？
- 最大步数？
- 超时？
- 人工干预？

### 4.2 Agent产品的PRD写法

**模块一：Agent能力描述**

```
Agent名称：智能客服Agent
能力：
- 能做什么：自动处理工单分类、查询订单、回复常见问题
- 不能做什么：不能处理退款（需人工）、不能处理投诉（需人工）
- 自主程度：半自主（查询和回复自动执行，退款和投诉转人工）
```

**模块二：Agent工作流**

```
用户提问
  → Agent判断意图
    → 简单问题 → 直接回复
    → 需要查询 → Function Calling → 生成回复
    → 复杂问题 → Plan-and-Solve → 多步执行 → 生成回复
    → 超出能力 → 转人工
```

**模块三：Agent状态管理**

```
- 对话历史保留：7天
- 用户偏好记忆：永久（存储在用户档案中）
- 工作记忆：当前会话内
```

**模块四：Agent终止条件**

```
- 最大思考步数：5步
- 最大执行时间：30秒
- 自动转人工条件：置信度<60% 或 用户明确要求
```

### 4.3 跟工程师沟通的常用话术

**讨论Agent设计时：**

> "这个Agent的自主程度怎么定？查询类操作可以全自动，但涉及金钱的操作需要人工确认。"

**讨论终止条件时：**

> "Agent最多允许走几步？如果5步还没解决，是直接转人工还是继续尝试？"

**讨论调试时：**

> "Agent每一步的Thought能不能在后台记录下来？我需要分析Bad Case的时候知道Agent是怎么想的。"

## 五、研发如何实现：Agent的工程架构

### 5.1 Agent框架

目前主流的Agent框架：

| 框架 | 特点 | 适用场景 |
|------|------|---------|
| LangGraph | 图结构，状态管理强 | 复杂多步Agent |
| AutoGen | 多Agent协作 | 多Agent系统 |
| CrewAI | 角色化Agent团队 | 多Agent协作 |
| Semantic Kernel | Microsoft出品，.NET生态 | 企业级应用 |
| LlamaIndex Agents | 专注数据检索的Agent | RAG+Agent |

**产品经理需要了解什么？**

不需要了解框架的代码细节，但需要理解框架提供的能力：

1. **状态管理**：框架是否支持Agent的长期记忆？
2. **工具集成**：框架是否方便接入Function Calling？
3. **调试能力**：框架是否提供Trace日志和可视化？
4. **多Agent支持**：框架是否支持多个Agent协作？

### 5.2 Agent的执行引擎

```
┌───────────────────────────────────────┐
│           Agent执行引擎                 │
│                                        │
│  ┌─────────────────────────────────┐  │
│  │         主循环（Main Loop）       │  │
│  │                                 │  │
│  │  WHILE not terminated:          │  │
│  │    1. 读取当前状态               │  │
│  │    2. 发送给模型（Thought）      │  │
│  │    3. 模型返回Action             │  │
│  │    4. 执行Action                 │  │
│  │    5. 获取Observation           │  │
│  │    6. 更新状态                   │  │
│  │    7. 检查终止条件               │  │
│  └─────────────────────────────────┘  │
│                                        │
│  ┌─────────────────────────────────┐  │
│  │         终止检测器                │  │
│  │    - 任务完成？                   │  │
│  │    - 达到最大步数？               │  │
│  │    - 超时？                      │  │
│  │    - 需要人工干预？               │  │
│  └─────────────────────────────────┘  │
└───────────────────────────────────────┘
```

### 5.3 Agent的日志和监控

```
Agent日志包含：
- 用户输入
- 每轮的Thought（推理过程）
- 每轮的Action（调用的工具）
- 每轮的Observation（工具返回结果）
- 最终输出
- 执行耗时
- Token消耗

监控指标：
- Agent执行成功率
- 平均执行步数
- 平均执行时间
- 转人工率
- 用户满意度
```

## 七、真实案例：客户工单系统的Agent升级

### 7.1 从"单步工具调用"到"Agent"

**第一阶段：单步工具调用（第五章）**

```
用户："查一下我的工单TK-12345"
→ 调用query_ticket(ticket_id="TK-12345")
→ 返回工单信息
→ 生成回复
```

问题：如果用户说"帮我查一下最近5个工单，把还没解决的优先级排一下"，单步调用搞不定。

**第二阶段：ReAct Agent（本章）**

```
用户："帮我查一下最近5个工单，把还没解决的优先级排一下"

Round 1:
Thought: 用户需要查询最近工单 → 调用search_tickets(user_id=xxx, limit=5)
Action: search_tickets(user_id="U001", limit=5)

Round 2:
Observation: 返回5个工单，其中3个未解决
Thought: 需要按优先级排序 → 调用sort_by_priority(tickets=[...])
Action: sort_by_priority(tickets=[TK-100, TK-200, TK-300])

Round 3:
Observation: 排序结果：TK-200(高), TK-100(中), TK-300(低)
Thought: 可以生成回复了
Action: generate_reply("您的最近5个工单中，3个未解决...")

回复："您的最近5个工单中，有3个尚未解决：
1. TK-200（高优先级）- 登录问题
2. TK-100（中优先级）- 功能建议
3. TK-300（低优先级）- 账号问题
需要我详细介绍某个工单吗？"
```

**第三阶段：带Reflection的Agent（进阶）**

```
在生成回复之前，增加一步：

Reflection:
Thought: 让我检查一下回复是否完整：
- 是否包含了所有未解决的工单？是（3个）
- 是否标注了优先级？是（高/中/低）
- 是否提供了下一步操作建议？是（"需要我详细介绍吗？"）
- 语气是否友好？是

回复质量检查通过 → 发送
```

### 7.2 客户工单Agent的设计参数

```
Agent配置：
- 最大步数：5步
- 超时时间：30秒
- 终止条件：
  - 任务完成 → 停止
  - 达到5步 → 转人工
  - 超时30秒 → 转人工
- 人工介入点：
  - 涉及退款 → 必须人工确认
  - 涉及投诉 → 必须人工确认
  - 置信度<60% → 转人工
- 记忆范围：
  - 短期记忆：当前对话历史
  - 长期记忆：用户偏好和历史工单
```

## 八、常见误区

### 误区一："所有场景都需要Agent"

**真相**：Agent有额外的延迟和成本。简单任务用单步Prompt就够了，不需要Agent。

**实践方法**：用复杂度评估——一问一答用Prompt，多步推理用Agent。

### 误区二："Agent越自主越好"

**真相**：自主性越高，不可预测性越强。在高风险场景（如金融、医疗），需要限制Agent的自主性。

**实践方法**：根据场景的风险等级设定Agent的自主程度。

### 误区三："Agent不会犯错误"

**真相**：Agent会犯错误——选错工具、提取错参数、过度推理（明明一步能解决偏要走三步）。

**实践方法**：设置最大步数限制和超时机制，防止Agent"想太多"。

### 误区四："Agent的记忆越多越好"

**真相**：记忆越多，Token消耗越大，成本越高。而且不是所有记忆都有用。

**实践方法**：选择性记忆——只记住对用户有价值的信息（偏好、历史问题、关键决策）。

### 误区五："Agent调试和普通软件一样"

**真相**：Agent的行为是概率性的，同样的输入可能产生不同的执行路径。传统调试方法不够用。

**实践方法**：建立Agent的Trace日志系统，记录每一步的Thought、Action、Observation。

### 误区六："ReAct就是让模型多思考几步"

**真相**：ReAct不仅仅是"多思考"，而是"思考-行动-观察"的闭环。每行动一步，都基于上一步的观察来调整下一步。

**实践方法**：确保Agent的每一轮Thought都能访问到上一轮的Observation。

### 误区七："Agent上线后就结束了"

**真相**：Agent的行为会随着工具变化、模型更新、用户行为变化而演化。需要持续监控和调整。

**实践方法**：建立Agent效果监控体系，跟踪执行步数、转人工率、用户满意度。

## 九、面试题

### 初级

**Q1：什么是Agent？它和普通的AI功能有什么区别？**

**参考答案：**

Agent是一个能自主感知、思考、行动的系统。它和普通的AI功能的核心区别在于：

- 普通AI功能：用户问一句，AI答一句。一次交互完成一个任务。
- Agent：用户提一个目标，Agent自己决定需要做什么、怎么做、按什么顺序做。

Agent有记忆（能记住之前的步骤）、有工具（能调用外部API）、有推理能力（能决定下一步做什么）。

---

**Q2：ReAct是什么意思？**

**参考答案：**

ReAct = Reasoning（推理）+ Action（行动）。

它是一种Agent架构，让Agent在思考和行动之间交替进行：
1. 思考：我现在知道了什么？下一步该做什么？
2. 行动：调用工具执行
3. 观察：工具返回了什么？
4. 回到思考步骤

这种方式比"先想完再做"或"直接做不想"都更灵活、更可靠。

---

### 中级

**Q3：Agent的终止条件怎么设计？如果Agent卡在无限循环里怎么办？**

**参考答案：**

终止条件的设计：

1. **任务完成**：Agent判断目标已达成
2. **最大步数**：限制Agent最多执行N步（通常5-10步）
3. **超时**：执行时间超过阈值（如30秒）
4. **人工干预**：用户说"停"或Agent请求人工确认

防止无限循环：

- 最大步数是最有效的防循环机制
- 记录已执行的步骤，如果发现重复步骤就终止
- 设置超时时间，超过时间强制终止
- 在Prompt中明确告诉Agent"最多执行N步"

---

**Q4：如何设计一个Agent的记忆系统？**

**参考答案：**

记忆系统分三层：

1. **短期记忆**：当前对话的历史记录。限制在最近N轮对话，受上下文窗口约束。

2. **工作记忆**：Agent在执行任务过程中的中间结果。比如已经调用了哪些工具、得到了什么结果。存储在Agent的内部状态中，一个会话结束后清除。

3. **长期记忆**：跨会话的记忆。存在向量数据库或关系数据库中。包括用户偏好、历史关键信息。每次新会话开始时，检索相关的长期记忆注入到短期记忆中。

---

### 高级

**Q5：你设计了一个客服Agent，发现它有时"想太多"——明明一步能解决的问题走了五步。你怎么办？**

**参考答案：**

这个问题叫"过度推理"，常见原因和解决方案：

**原因一：Prompt没有明确限制**
- 解决：在系统提示词中加入"如果一步能解决，不要分多步"

**原因二：Agent没有"任务完成"的判断标准**
- 解决：在Prompt中定义"什么算完成任务"

**原因三：模型本身倾向于多步推理**
- 解决：降低Temperature参数，减少随机性

**原因四：工具返回的信息不足以让Agent判断任务完成**
- 解决：优化工具返回格式，让Agent能明确判断"够了"

**系统性方案：**
1. 分析Agent的Trace日志，找出"想太多"的典型模式
2. 针对性优化Prompt和工具返回格式
3. 设置最大步数限制作为兜底
4. 建立"步数-效果"评估，找到最优步数

---

**Q6：设计一个多步推理的Agent产品，需要考虑哪些关键设计点？**

**参考答案：**

**1. 任务分解**
- Agent如何把复杂任务分解成子任务？
- 子任务之间是串行还是并行？

**2. 工具设计**
- 每个子任务需要调用哪些工具？
- 工具的输入输出格式是否一致？

**3. 状态管理**
- Agent如何记住中间结果？
- 如何恢复中断的会话？

**4. 错误处理**
- 某一步失败了怎么办？重试？换方案？转人工？
- 工具超时怎么处理？

**5. 终止条件**
- 什么时候算"完成了"？
- 最大步数和超时时间怎么设？

**6. 可观测性**
- 如何记录Agent的每一步决策？
- 如何调试和复盘Agent的行为？

**7. 用户体验**
- 是否展示Agent的思考过程？
- 用户能否中途干预？
- 等待时间过长时如何反馈进度？

## 十、本章总结

本章我们系统学习了Agent的核心架构：

1. **Agent的四件套**：大脑（模型）+ 记忆（短期/工作/长期）+ 工具（Function Calling/RAG/API）+ 行动（执行和反馈）
2. **ReAct范式**：思考-行动-观察的循环，比"全想再做"或"全做不想"都好
3. **Plan-and-Solve**：复杂任务先规划再执行，支持自我规划和思维树
4. **Reflection模式**：完成任务后自我检查，减少幻觉和提高质量
5. **状态管理**：短期记忆（对话历史）、工作记忆（中间结果）、长期记忆（跨会话）
6. **终止条件**：任务完成、最大步数、超时、人工干预
7. **客户工单Agent升级**：从单步调用到ReAct Agent再到带Reflection的Agent
